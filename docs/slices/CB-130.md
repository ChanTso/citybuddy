## Goal

Add owner-scoped bounded conversation memory with monotonic summary watermarks, construct
role-separated prompts from authoritative and explicitly untrusted inputs, apply deterministic PII
minimization, and enforce risk-tiered output safety without weakening receipt, citation, identity,
budget, or durable evidence truth.

## In scope

- Persist one owner/session-scoped summary record in `cs_db` with a monotonic
  `summary_until_turn` watermark, source-turn commitment, bounded summary content, model/prompt policy
  version, and immutable generation evidence. Support Redis may hold only a TTL-bound hot copy keyed
  by session and watermark; MySQL remains truth.
- Generate summaries only from a bounded contiguous prefix of durable owner-scoped turns and accepted
  evidence. A summary worker uses conditional compare-and-set so an older, duplicated, delayed, or
  restarted result cannot overwrite a newer watermark. Missing/evicted/malformed cache state rebuilds
  only from `cs_db`.
- Build each model prompt through a fixed budgeted `MemoryPacker`: versioned `SYSTEM` policy, closed
  ToolSpec definitions, current `USER` input, bounded recent turns, monotonic summary, allowlisted
  commerce-owned read-only CRM view, `UNTRUSTED RETRIEVED` public evidence, and `UNTRUSTED TOOL DATA`
  remain structurally separated. Untrusted content cannot become system/tool instructions.
- Minimize PII before model/provider boundaries and before optional observability: allow only the
  fields required by the current tool/prompt role, redact configured direct identifiers from memory
  and diagnostic payloads, and retain durable private truth only in its owning database. Redaction
  never fabricates business or receipt state.
- Apply server-owned risk tiers to output. Ordinary knowledge/chitchat uses the existing bounded
  streaming buffer; retrieved answers retain sufficiency/citation gates; sensitive-action success is
  rendered only from stored ActionReceipt projection; high-risk or policy-blocked content returns a
  bounded safe outcome. A secondary text/tool consistency guard may block contradictions but cannot
  create action truth.
- Version prompt, memory, redaction, and output-policy definitions in code and durable evidence so
  replay uses the recorded policy/evidence rather than silently applying a future prompt to history.
- Preserve the single ReAct mainline, shared attempt budget, provider-scoped circuits, OBO and
  ToolSpec authority, retrieval evidence, filtered SSE, receipt turn commit, and historical evidence.
- Add only exact `agent_app` migration/runtime grants for summary/policy evidence and exact Support
  Redis key access. No direct commerce database, Elasticsearch write, or cross-service truth access.

## Acceptance criteria

1. Summary input is an exact bounded contiguous prefix of durable turns owned by one user/session.
   Unknown, cross-owner/session, missing, duplicate, non-contiguous, over-bound, or contradictory
   source evidence fails closed without a new summary.
2. `summary_until_turn` only increases. Concurrent, delayed, duplicate, reordered, and restarted
   summarization cannot overwrite a newer watermark; the winner's source commitment and policy
   version match the exact summarized prefix.
3. Summary persistence is durable in `cs_db`. Support Redis entries have positive bounded TTL,
   include session/watermark/content commitment, and are accepted only when they exactly match
   MySQL; eviction, outage, stale value, malformed value, or cross-session key cannot override truth.
4. Prompt construction enforces one deterministic total budget before materialization and assigns
   each input to exactly one role: `SYSTEM`, `TOOLS`, `USER`, trusted bounded memory/CRM, `UNTRUSTED
   RETRIEVED`, or `UNTRUSTED TOOL DATA`. Retrieved/tool/user text cannot inject or replace system
   policy, ToolSpec, identity, scope, action confirmation, receipt, or budget controls.
5. The CRM view is read-only, owner-scoped, field-allowlisted, and fetched through the existing
   commerce OBO boundary. The model/client cannot select owner, fields, SQL, scope, or another
   session's profile.
6. PII rules are deterministic, versioned, and applied before model and optional trace boundaries.
   Fixed direct-identifier fixtures, nested untrusted content, Unicode/control forms, and structured
   tool data prove minimization without leaking raw private values into prompts, logs, errors, cache,
   or observability payloads.
7. Output tier is server-derived from durable route/tool/retrieval/receipt evidence. Caller/model
   labels cannot lower risk. Ordinary, retrieval-grounded, insufficient, sensitive-receipted,
   sensitive-unreceipted, contradictory, and blocked outputs follow distinct fixed behavior.
8. `action_receipt` remains the sole action-status carrier. Stored receipt projection may produce the
   fixed receipt event; token prose cannot create or replace it. The bounded known-claim lexicon and
   consistency guard remain defense in depth and do not claim complete natural-language
   classification.
9. Citation-bearing answers cite only allowlisted stored retrieval evidence actually used by the
   turn. Summary, CRM, untrusted tool data, future index contents, and regenerated text cannot invent
   or rewrite historical sources.
10. Model/provider timeout, malformed output, budget exhaustion, circuit rejection, redaction
    failure, cache/CRM/retrieval unavailability, and contradictory evidence cannot become a grounded
    answer, confirmed action, or successful terminal decision.
11. Prompt/memory/output policy version and bounded evidence needed for replay are durable. Replay of
    a completed historical turn performs no new business action and does not silently rewrite its
    receipt, citation, model-tier, or source truth.
12. Exact grants isolate summary/evidence tables and Support Redis keys. `agent_app` cannot access
    `commerce_db`, auth secrets, Redis admin/truth keys, or Elasticsearch writes; other runtime
    identities cannot mutate agent summary truth.
13. No authoritative handoff ticket, `HUMAN_PENDING` transition, failure-candidate export, web UI,
    deployment, real-provider credential, trace authority, or measured performance claim is
    introduced.

## Rejection paths

- Reject cross-owner/session turns or CRM, non-contiguous or unbounded history, stale watermark
  overwrite, source-commitment mismatch, malformed cache accepted as truth, and summary generation
  from model memory rather than durable evidence.
- Reject prompt concatenation that collapses trusted and untrusted roles, materializes unbounded
  history before applying limits, lets retrieved/tool/user content alter system policy/ToolSpec, or
  lets the model choose identity, scope, risk tier, receipt status, or action confirmation.
- Reject PII allowlists that omit nested fields or alternate encodings, redaction after provider/log
  transmission, raw private values in errors/traces/cache, and redaction that changes business truth
  or commitments.
- Reject caller/model-selected lower risk, ungrounded citations, insufficient retrieval rendered as
  supported, token prose treated as action truth, synthetic/mutated receipt events, multiple terminal
  outputs, and post-commit retry that executes commerce again.
- Reject cache, Elasticsearch, optional tracing, or current model output as authority over MySQL
  summary/receipt/business truth or stored retrieval evidence.
- Reject broad grants, cross-database joins, direct private-store access, arbitrary prompt/provider
  configuration, unversioned policy mutation, and CB-131 or later work implemented early.

## Out of scope

- commerce PendingAction/ActionReceipt mutation owned by CB-120 and agent receipt projection/turn
  commit owned by CB-121;
- new retrieval/index ranking, rebuild rollback, retired-index cleanup, or changes to the verified
  CB-113 forward alias contract;
- authoritative handoff tickets, SLA/Outbox and `HUMAN_PENDING` behavior owned by CB-131;
- reviewed failure-candidate capture/export owned by CB-132;
- web UI, observability implementation, deployment, real-provider access, scripted demos/fault
  drills, and measured load/latency/quality.

## Required evidence

- real MySQL concurrent/reordered/restart summary evidence proving exact source-prefix commitment,
  monotonic CAS watermark, rollback, bounds, ownership, policy versioning, and no stale overwrite;
- real Support Redis TTL/eviction/outage/malformed/stale/cross-session fault evidence proving MySQL
  truth and exact key/grant isolation;
- executable prompt-structure and acquisition-bound tests for every role, deterministic budget/order,
  hostile instruction fixtures in user/retrieved/tool data, and inability to alter system policy,
  ToolSpec, identity, scope, receipt, or risk tier;
- deterministic PII corpus across structured/nested text, Unicode/control forms, logs/errors/cache,
  model payloads, and optional no-op observability payloads, with exact allowed-field evidence and no
  raw private values beyond owning durable truth;
- output matrix for ordinary, grounded, insufficient, receipted sensitive, unreceipted/contradictory
  sensitive, blocked, malformed, unavailable, and exhausted-budget paths across streaming and
  non-streaming responses;
- replay and restart evidence proving immutable receipt/citation/source/policy truth, explanation-only
  retry, no second commerce execution, and unchanged CB-080/081/082/091/121 behavior;
- exact MySQL/Support Redis grants and cross-database/private-store/Elasticsearch-write denial;
- evidence that no handoff/ticket, export, web, deployment, provider credential, observability
  authority, performance, rollback, cleanup, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
