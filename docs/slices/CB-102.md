## Goal

Expose bounded, evaluation-profile-only commerce state, audit-reference, and version views for one active or terminal CB-101 sandbox, using only authoritative `commerce_db` truth and server-owned capability metadata, without exposing agent evidence, implementing an evaluator, or changing business behavior.

## In scope

- Add evaluation-profile-only `GET /api/eval/state` behind the independent evaluation API credential. Require exact sandbox context and return a bounded allowlisted snapshot of the sandbox registry and participating commerce fixture/business truth created or exercised through CB-101.
- Keep `commerce_db` authoritative. State reads use the exact sandbox predicate on every participating table, preserve current durable statuses and versions, and never infer completion from Redis, Elasticsearch, RocketMQ delivery, caches, logs, or client claims.
- Add evaluation-profile-only `GET /api/eval/audit/{sessionId}` behind the same management boundary. Require the requested sandbox and return only bounded commerce-owned audit/reference records whose persisted sandbox and support-session association both match.
- Add the minimum append-only commerce evaluation audit-reference record needed to associate an evaluation synchronous operation with sandbox, support session, trace/idempotency correlation, authoritative business entity/type, and entity version. It contains references and bounded outcome metadata, not duplicated business payload or agent evidence, and commits atomically with the owning commerce mutation where a mutation occurs.
- Treat the audit record as a locator, not business truth. The referenced order, reservation, payment, refund, ledger, or later receipt remains authoritative in its owning commerce table; conflicting or missing referenced truth fails closed rather than being reconstructed from the audit row.
- Add evaluation-profile-only `GET /api/eval/version` behind the independent evaluation API credential. Return only server-owned build, schema-compatibility, and evaluation-capability identifiers from executable configuration and startup-validated migration state; never expose arbitrary environment, dependency, host, database, or credential metadata.
- Define bounded, deterministic ordering, pagination/limits, response schemas, and error schemas for all three routes. Client-selected field lists, table names, SQL fragments, joins, sorts, capability overrides, or unbounded scans are not accepted.
- Keep the routes read-only from the evaluator's perspective. Repeated reads do not mutate business truth, advance the sandbox lifecycle, extend TTL, invoke tools, publish messages, trigger callbacks, or create an evaluation result.
- Extend only exact `commerce_app` migration/runtime privileges needed for the append-only audit-reference record and allowlisted reads. `auth_app` and `agent_app` remain denied; `commerce-service` does not read `cs_db` or auth-private records.
- Use real MySQL, evaluation-profile authentication, sandbox isolation, restart, bounds, redaction, and exact-grant evidence. Do not add the CB-103 agent evidence endpoint, CB-104 asynchronous liveness guards, CB-105 sandbox callback behavior, ServiceEval, web, deployment, or later work.

## Acceptance criteria

1. `/api/eval/state`, `/api/eval/audit/{sessionId}`, and `/api/eval/version` are loaded only in the evaluation profile and require the independent evaluation API credential; production returns route-not-found and does not treat a user JWT, service credential, sandbox header, or management body field as an authentication fallback.
2. State requires one bounded sandbox identifier and returns only the allowlisted registry and commerce fixture/business families owned by that sandbox, with current durable status/version fields and deterministic bounded ordering. Every SQL path contains the exact sandbox predicate.
3. Unknown, malformed, cross-sandbox, expired-inconsistent, or unauthorized state requests fail closed without revealing whether another sandbox or user owns matching data. A terminal `DEAD` sandbox remains observable only as explicitly allowed historical truth and cannot be reopened or mutated by the read.
4. Audit requires exact sandbox plus support-session association and returns only bounded append-only references persisted for that pair. Cross-sandbox, cross-session, unknown, malformed, conflicting, or unassociated requests reject without falling back to sandbox-only or session-only lookup.
5. Each commerce audit reference has a unique idempotent identity and stores only bounded sandbox/session/trace correlation, business entity/type/id/version, outcome/reference metadata, and time. It commits atomically with its owning commerce mutation when applicable; rollback leaves neither the mutation nor a false audit reference.
6. Audit references never replace or copy authoritative business payload. Reads validate the referenced commerce row and sandbox/version association where current truth is required; missing, mismatched, or conflicting references fail closed rather than returning reconstructed success.
7. Version returns a fixed allowlist of build, schema-compatibility, and evaluation-capability identifiers derived from executable server configuration and startup-validated migration state. Repeated calls are deterministic for one deployed build and reveal no secret, host, environment dump, database details, provider identifiers, or arbitrary dependency inventory.
8. Request and response bounds prevent unbounded table scans or payloads. Pagination/limit rules, stable ordering, maximum record counts, identifier lengths, and error schemas are server-owned; unknown fields and caller-selected tables, columns, filters, joins, sorts, or capability versions reject before SQL.
9. The three evaluator reads do not mutate sandbox/business state, extend expiry, provision/revoke identities, execute tools, publish messages, invoke payment callbacks, rerun retrieval, or create evaluator verdict truth. Restart and repeated reads return the same durable view subject only to later authoritative commerce commits.
10. Exact-grant evidence proves only required `commerce_app` reads and append-only audit-reference DML, commerce migration ownership, `auth_app`/`agent_app` denial, runtime-DDL denial, and no commerce runtime access to `cs_db` or auth-private tables.
11. Real integration proves evaluation authentication/profile isolation, active and dead sandbox views, bounded multi-family state, session-bound audit references, atomic rollback, cross-sandbox/session rejection, deterministic version output, restart persistence, redaction, bounds, permissions, and absence of CB-103/104/105 or evaluator behavior.

## Rejection paths

- Reject production-profile routes; missing, invalid, wrong-purpose, or substituted evaluation credentials; and any user JWT, service credential, sandbox header, or anonymous fallback used as management authentication.
- Reject missing/malformed sandbox or session identifiers, cross-sandbox/session lookup, sandbox-only audit fallback, client-selected owner/correlation truth, and errors that disclose another sandbox's existence or data.
- Reject unscoped SQL, optional sandbox predicates, broad joins, full-table scans, caller-selected tables/columns/filters/sorts, arbitrary SQL or metadata queries, unbounded limits, and nondeterministic pagination.
- Reject state synthesized from Redis, Elasticsearch, RocketMQ delivery, logs, caches, model text, or client claims; reject audit rows treated as replacements for their authoritative commerce records.
- Reject mutable/overwritten audit references, duplicate conflicting identities, audit success committed without its owning business mutation, copied private payloads, raw request/response bodies, credentials, tokens, prompts, provider details, or database errors.
- Reject version output containing environment dumps, host identity, filesystem paths, credentials, internal endpoints, arbitrary dependency inventories, raw migration SQL, or caller-selected capability/version values.
- Reject a read that activates/completes/reopens a sandbox, extends TTL, provisions/revokes identity, invokes a tool/callback, publishes an event, reruns agent/retrieval work, or records an evaluator verdict.
- Reject CB-103 agent evidence access, CB-104 asynchronous liveness closure, CB-105 sandbox callback behavior, ServiceEval, web, deployment, or later work implemented early.
- Reject capability claims based only on handler mocks without real `commerce_db`, evaluation profile/authentication, sandbox/session association, atomic audit reference, restart, bounds, redaction, and permission evidence.

## Out of scope

- agent-owned `cs_db` evidence retrieval and `GET /api/eval/evidence/{traceId}`;
- asynchronous consumer sandbox-liveness checks, inactive-work drop/archive behavior, and post-completion message closure;
- sandbox-bound mock-payment callback extensions, callback-triggered evaluation shortcuts, or real external payment effects;
- ServiceEval, evaluator verdict/scoring storage, direct evaluator database access, production evaluation routes, web UI, deployment, and performance claims;
- changes to sandbox identity/provisioning, reset/activation/completion/janitor semantics, transaction truth, retrieval, actions/receipts, memory, handoff, or knowledge synchronization.

## Required evidence

- executable OpenAPI/profile/contract tests for all three routes, independent evaluation credential, route absence in production, bounded schemas, stable ordering/pagination, unknown fields, and public error/redaction allowlists;
- real `commerce_db` evidence for exact sandbox predicates across every state family, active/dead historical behavior, cross-sandbox isolation, bounded reads, current durable truth, and no cache/projection substitution;
- real append-only audit-reference migration and transaction evidence for sandbox/session/trace/entity association, same-intent identity, conflicting reuse, atomic commit/rollback, authoritative-row validation, restart persistence, and no copied private payload;
- deterministic version evidence for fixed build/schema/capability identifiers, startup compatibility validation, stable repeat/restart output, caller-override rejection, and forbidden environment/host/credential/database metadata;
- exact `commerce_app` grant evidence plus `auth_app`/`agent_app`, runtime-DDL, `cs_db`, and auth-private denial;
- evidence that reads cause no lifecycle/business mutation, message publication, callback, tool/retrieval execution, or evaluator result, and that no CB-103/104/105, ServiceEval, web, deployment, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
