## Goal

Establish direct-user standard ordering with user-scoped idempotency and an atomic MySQL stock update, preserving commerce ownership and producing deterministic, auditable rejection outcomes without allowing retries, Redis, or messaging to become transaction truth.

## In scope

- Add commerce-owned migrations in `commerce_db` for standard orders, order lines or an equivalent immutable order snapshot, and the constraints required for user-scoped request idempotency, using only the existing commerce migration and runtime identities.
- Define executable OpenAPI and contract tests for `POST /api/orders`. Accept only the CB-020 direct-user JWT chain, derive the order owner from the validated subject, require the route permission and `Idempotency-Key`, and reject request-body identity substitution.
- Validate the product identifier, positive bounded quantity, published and available state, current product/version expectation, and sufficient stock against authoritative MySQL state. Capture the server-authoritative product and price snapshot used for the order; never trust client price or stock.
- Atomically perform a conditional stock decrement or optimistic version check, create the owned standard order, persist the idempotency result, and add the uniquely identified order Outbox record in one MySQL business transaction.
- Scope an idempotency key to the validated user and canonical order intent. A repeat with the same intent returns the existing order without another stock change; the same key with different intent rejects deterministically.
- Handle only explicitly recognized transient database concurrency failures with a configured finite retry bound. Each retry revalidates authoritative state; exhaustion and unexpected database or configuration errors fail visibly without a false order result.
- Return stable rejection categories and correlation evidence for authentication, authorization, validation, ownership, idempotency conflict, stale version, insufficient stock, and exhausted concurrency without exposing credentials or private data.
- Prove least-privilege separation: `commerce_app` receives only the DML required for the new commerce families, while `auth_app` and unrelated identities cannot read or mutate order or inventory truth.

## Acceptance criteria

1. Commerce migrations create stable order identity, token-subject ownership, immutable order-line or equivalent product/price snapshots, user-scoped idempotency uniqueness, and the stock/version constraints needed for atomic creation without widening unrelated grants.
2. A valid direct-user request with a unique `Idempotency-Key`, current product version, legal quantity, published and available product, and sufficient stock creates one owned standard order and decrements MySQL stock exactly once.
3. Order creation, conditional stock mutation, idempotency result, and uniquely identified order Outbox row commit or roll back together; no cache, message acknowledgement, or request payload is accepted as transaction or inventory truth.
4. Repeating the same user's key with the same canonical intent returns the same order and leaves stock and Outbox cardinality unchanged. A different intent under that user's key rejects; another user's same literal key is resolved only inside that other user scope and cannot expose or mutate the original order.
5. Missing or invalid authentication, wrong issuer/audience/type, missing permission, request-body owner substitution, illegal quantity, client-supplied price/stock authority, missing/unpublished/unavailable product, stale version, and insufficient stock all reject before a successful order result; transaction evidence proves no partial stock, order, idempotency, or Outbox commit.
6. Concurrent requests against limited stock cannot oversell. Conditional update or optimistic-version conflicts retry only for the recognized transient cases and only to the configured finite bound, re-reading current MySQL state on each attempt.
7. A controlled retryable conflict can succeed within the bound; controlled exhaustion returns a stable failure with no false success, while an unrecognized database or programmer/configuration error is not swallowed or converted into a retryable business rejection.
8. Rejection results carry a stable category and safe correlation evidence sufficient to distinguish authentication, authorization, validation, ownership, idempotency, stale-version, insufficient-stock, and retry-exhaustion paths without leaking credentials or another user's order data.
9. Permission tests prove `commerce_app` has only required runtime DML on the new order and inventory families, `auth_app` is denied access, runtime identities cannot perform DDL, and migration/bootstrap identities are absent from application configuration.
10. No cart, seckill activity/reservation, Redis quota or admission, inventory ledger, delayed cancellation, payment/refund, Agent tool, evaluation sandbox, or web UI behavior is introduced.

## Rejection paths

- Reject an order owner, price, stock, availability, publication state, or successful result inferred from request data, Redis, RocketMQ, Elasticsearch, or another non-authoritative projection.
- Reject an idempotency implementation that scopes only by a global client key, decrements stock before duplicate resolution, returns another user's result, or treats a conflicting payload as the original request.
- Reject a transaction that can leave stock decremented without its order and idempotency result, an order committed without its required Outbox row, or any partial record after a rejected request.
- Reject a read-then-write stock check that can oversell, an unbounded or catch-all retry loop, fixed sleeps used as correctness, or a retry that reuses stale product/inventory state.
- Reject authentication fallback, body-identity substitution, cross-user disclosure, permissive token validation, or a rejection response/log containing credentials or private order data.
- Reject capability claims based only on mocks without real MySQL concurrency, rollback, idempotency, permission-denial, Outbox, and controlled retry-exhaustion evidence.

## Out of scope

- Cart and multi-step checkout behavior beyond the single standard-order contract;
- seckill quota allocation, Redis Lua admission, transaction messages, reservations, and seckill order consumption;
- inventory-ledger movements, delayed unpaid cancellation, payment, refund, and reconciliation;
- Agent ToolSpecs, OBO order tools, sensitive prepare/confirm actions, evaluation sandbox behavior, and web UI;
- production deployment, broad resilience, and measured performance or operational-readiness claims.

## Required evidence

- migration and permission output for order, stock, idempotency, and Outbox ownership, including explicit `auth_app` denial and runtime DDL denial;
- executable OpenAPI/contract tests for a valid direct-user order and every listed identity, validation, ownership, idempotency, stale-version, and insufficient-stock rejection;
- real MySQL transaction evidence showing stock, order, idempotency result, and Outbox atomic commit plus rollback with no partial rows;
- same-key/same-intent replay, conflicting-intent rejection, cross-user isolation, and concurrent duplicate evidence proving one result and one stock mutation;
- real constrained-stock concurrency evidence proving no oversell;
- controlled recognized-transient retry success, finite retry exhaustion, and unrecognized-error visibility with the configured bound recorded;
- evidence that Redis, RocketMQ acknowledgement, and Elasticsearch do not establish or repair order/inventory truth and that no later business slice was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
