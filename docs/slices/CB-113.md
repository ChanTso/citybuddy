## Goal

Rebuild the complete public-knowledge projection into a new versioned Elasticsearch physical index, prove that it represents one complete authoritative public-source snapshot plus every accepted concurrent incremental change, and atomically switch the stable read alias only after all structural, corpus, retrieval, and convergence gates pass, while retaining the prior index for bounded controlled rollback.

## In scope

- Add one bounded `knowledge-indexer` rebuild command boundary that creates a fresh server-named `knowledge_docs_vN` physical index. Callers cannot select an arbitrary index, alias, mapping, corpus, validation result, source version, cutoff, or cleanup target.
- Build from bounded published-source snapshot payloads supplied by the owning source boundaries. The snapshot carries stable source identities, source versions, publication/tombstone state, bounded public content, a server-generated snapshot identity and cutoff/watermark, row count, and a deterministic content commitment. `knowledge-indexer` does not read `commerce_db` or `cs_db`, and the current Elasticsearch projection is not promoted into source truth.
- Keep snapshot acquisition and replay compatible with the CB-111 production-only event path. Buffer or durably observe accepted source updates after the snapshot cutoff, load the complete snapshot, replay every newer publication/tombstone in source-version order, and repeat the bounded catch-up check until the candidate is proven current at one explicit handoff watermark.
- Create the candidate with the exact approved mapping, analyzer, dense-vector dimensions, strict public metadata allowlist, deterministic document/chunk identities, and source-version/tombstone semantics already fixed by CB-090 and CB-111. Never write rebuild documents through the read alias.
- Validate the candidate before exposure: exact mapping/settings, committed source set and count, per-source latest version/tombstone state, absence of duplicates/private fields, full indexing completion, shard health, and fixed BM25/dense/hybrid retrieval probes including negative public-boundary checks.
- Treat every validation response as untrusted runtime evidence. Timeout, partial shards, task failure, count/digest mismatch, malformed payload, version gap/conflict, stale catch-up, or unavailable dependency fails closed and leaves the read alias unchanged.
- Switch `knowledge_docs_read` with one Elasticsearch alias action that removes the one current approved target and adds the one validated candidate. Verify the resulting alias resolves to exactly that candidate; an ambiguous, missing, concurrently changed, or unexpected source alias aborts without claiming success.
- Retain the previous physical index under a bounded rollback lease. Rollback uses the same mapping, corpus, catch-up, health, and single-target validation gates and one atomic alias action; it cannot silently discard source updates accepted after the original switch.
- Keep cleanup separate from switch success and limited to server-recorded expired rebuild candidates or rollback predecessors after rollback authorization has expired. Never delete the active alias target, an unverified ownership target, or an index outside the exact versioned family.
- Preserve CB-090 retrieval, CB-091 rerank/sufficiency/evidence, CB-111 incremental ordering/tombstones, and CB-112 cache truth hierarchy. Historical retrieval evidence remains bound to the index version and sources actually used and is never rewritten by rebuild or rollback.
- Use the existing `knowledge-indexer` Elasticsearch ownership and production-only messaging boundary. Add no direct database access, Support Redis authority over rebuild truth, request-thread dual write, evaluation carrier, public FAQ authoring API, web, deployment, provider access, or later behavior.

## Acceptance criteria

1. A rebuild creates one fresh server-named physical index with the exact approved mapping/settings and never mutates through `knowledge_docs_read` or overwrites an existing physical index.
2. The candidate is loaded only from a complete owner-supplied public snapshot with stable identity, cutoff/watermark, count, and deterministic content commitment. Missing, duplicate, malformed, private, unpublished, stale, or conflicting source records fail closed.
3. Every CB-111 publication or tombstone accepted after the snapshot cutoff is applied to the candidate in source-version order before handoff. Concurrent updates in every snapshot/load/validate/switch window cannot be lost, resurrected, duplicated, or regressed.
4. Validation proves exact mapping/analyzer/vector compatibility, candidate health and task completion, committed source-set equality, latest per-source version/tombstone state, deterministic document identities, public-field allowlisting, and the fixed real retrieval probes.
5. Count, maximum version, boundary rows, or successful bulk HTTP status alone are never sufficient validation evidence. The complete source-set commitment and per-source convergence must match, and every Elasticsearch response must prove no timeout or partial shard/task failure.
6. The alias changes only through one atomic remove/add action from the exactly observed current target to the exactly validated candidate. On any precondition failure, the prior alias target remains the sole readable index and no success is reported.
7. After switch, repeated reads resolve exactly one candidate version and preserve the existing retrieval, rerank, sufficiency, citation, and durable evidence contracts. Historical reads retain their recorded index version and sources.
8. The old index remains available for a bounded rollback period. A rollback first catches the target up to a new explicit watermark, revalidates it, and atomically restores one alias target without discarding post-switch publications or tombstones.
9. Failed, abandoned, and superseded candidates remain non-readable and are cleaned only by bounded server-owned ownership/lease rules. Active, rollback-eligible, unknown, or out-of-family indexes cannot be deleted.
10. Elasticsearch outage, timeout, partial bulk/shard/task success, indexer restart, duplicate command, concurrent rebuild command, alias race, or crash before/after the alias action converges to an observable idempotent result without a multi-target alias or false completion.
11. Exact-access evidence proves `knowledge-indexer` has only the required Elasticsearch index/alias operations and no `commerce_db`, `cs_db`, runtime DDL/admin, Support Redis truth, or cross-database access. Source owners do not dual-write Elasticsearch.
12. Real integration covers complete rebuild, concurrent publication and tombstone catch-up in every handoff phase, commitment mismatch, version gap/conflict, malformed/private input, partial bulk/shard/task failure, validation probe failure, alias race, restart/crash windows, rollback catch-up, safe cleanup, and unchanged historical evidence.
13. No public authoring API, new ranking/sufficiency policy, cache-authoritative behavior, evaluator, action/receipt, memory, handoff, web, deployment, real-provider access, or measured performance claim is introduced.

## Rejection paths

- Reject caller-selected index/alias/corpus/cutoff/version/validation/cleanup values, reuse of an existing physical index, writes through the read alias, and an index outside the exact versioned family.
- Reject source snapshots derived only from the current Elasticsearch projection, direct indexer database reads, incomplete source sets, aggregate-only validation, commitment mismatch, duplicate identities, version gaps/conflicts, stale tombstones, private fields, and arbitrary metadata.
- Reject a handoff that does not capture and replay every accepted concurrent update through an explicit watermark, or that treats queue emptiness, wall-clock delay, bulk acceptance, count, or maximum version as proof of convergence.
- Reject missing, ambiguous, or concurrently changed aliases; separate non-atomic remove/add calls; switching before all gates pass; and reporting success without rereading exactly one target.
- Reject timeout, HTTP-success partial shards/tasks, malformed Elasticsearch responses, incompatible mapping/analyzer/vector shape, failed retrieval probes, redaction failure, or unavailable dependencies treated as validation success.
- Reject rollback to stale content, rollback without catch-up and full revalidation, unbounded predecessor retention, automatic deletion of the current target, wildcard cleanup, and cleanup based only on caller-supplied names.
- Reject rewriting historical retrieval evidence, making cache or Elasticsearch authoritative over published source truth, request-thread dual writes, direct cross-database access, broad admin grants, evaluation reachability inferred from mocks, or later-slice behavior implemented early.
- Reject capability claims based only on unit fakes without the pinned Elasticsearch runtime, real source snapshot commitment, concurrent CB-111 delivery, actual atomic alias actions, restart/crash recovery, rollback, and safe cleanup evidence.
