## Goal

Rebuild the complete public-knowledge projection into a fresh versioned Elasticsearch physical index, prove that it contains one complete authoritative public-source snapshot plus every accepted concurrent incremental change through an explicit handoff watermark, and atomically switch the stable read alias only after every validation gate passes.

## In scope

- Add one bounded `knowledge-indexer` rebuild command boundary that creates a fresh server-named `knowledge_docs_vN` physical index. Callers cannot select an arbitrary index, alias, mapping, corpus, validation result, source version, cutoff, or cleanup target.
- Build from bounded published-source snapshot payloads supplied by the owning source boundaries. The snapshot carries stable source identities, source versions, publication/tombstone state, bounded public content, a server-generated snapshot identity and cutoff/watermark, row count, and a deterministic content commitment. `knowledge-indexer` does not read `commerce_db` or `cs_db`, and the current Elasticsearch projection is not source truth.
- Keep snapshot acquisition and replay compatible with the CB-111 production-only event path. Buffer or durably observe accepted source updates after the snapshot cutoff, load the complete snapshot, replay every newer publication/tombstone in source-version order, and repeat the bounded catch-up check until the candidate is proven current at one explicit handoff watermark.
- Create the candidate with the exact approved mapping, analyzer, dense-vector dimensions, strict public metadata allowlist, deterministic document/chunk identities, and source-version/tombstone semantics fixed by CB-090 and CB-111. Never write rebuild documents through the read alias.
- Validate exact mapping/settings, committed source set and count, per-source latest version/tombstone state, absence of duplicates/private fields, complete indexing, shard health, and fixed BM25/dense/hybrid retrieval probes including negative public-boundary checks.
- Reconcile candidate `SYNC_EVENT` markers bidirectionally against the Broker journal events already covered by the owner snapshot. The stable event-id sets and every marker commitment field must be equal; shape-valid orphan or missing markers fail closed.
- Treat every validation response as untrusted runtime evidence. Timeout, partial shards, task failure, count/digest mismatch, malformed payload, version gap/conflict, stale catch-up, or unavailable dependency fails closed and leaves the read alias unchanged.
- Switch `knowledge_docs_read` with one Elasticsearch alias action that removes the one current approved target and adds the one validated candidate. Verify that the alias resolves to exactly that candidate; an ambiguous, missing, concurrently changed, or unexpected source alias aborts without claiming success.
- Record the predecessor and switch watermark under a bounded server-owned rollback lease for CB-114. This slice does not execute rollback or delete any candidate or predecessor.
- Preserve CB-090 retrieval, CB-091 rerank/sufficiency/evidence, CB-111 incremental ordering/tombstones, and CB-112 cache truth hierarchy. Historical retrieval evidence remains bound to the index version and sources actually used.
- Use the existing `knowledge-indexer` Elasticsearch ownership and production-only messaging boundary. Add no direct database access, Support Redis authority over rebuild truth, request-thread dual write, evaluation carrier, public FAQ authoring API, rollback execution, index cleanup, web, deployment, provider access, or later behavior.

## Acceptance criteria

1. A rebuild creates one fresh server-named physical index with the exact approved mapping/settings and never mutates through `knowledge_docs_read` or overwrites an existing physical index.
2. The candidate is loaded only from a complete owner-supplied public snapshot with stable identity, cutoff/watermark, count, and deterministic content commitment. Missing, duplicate, malformed, private, unpublished, stale, or conflicting source records fail closed.
3. Every CB-111 publication or tombstone accepted after the snapshot cutoff is applied to the candidate in source-version order before handoff. Candidate `SYNC_EVENT` markers exactly equal the owner-covered Broker journal event set by event id and full marker content; a missing, extra, or contradictory marker fails validation. The exact validated marker set is durably checkpointed before the first non-atomic Broker ACK, so validation-to-ACK arrivals remain unacknowledged and a partial-ACK restart reconstructs the acknowledged prefix together with the redelivered suffix. Concurrent updates in every snapshot/load/validate/switch window cannot be lost, resurrected, duplicated, or regressed.
4. Validation proves exact mapping/analyzer/vector compatibility, candidate health and task completion, committed source-set equality, latest per-source version/tombstone state, deterministic document identities, public-field allowlisting, and the fixed real retrieval probes.
5. Count, maximum version, boundary rows, or successful bulk HTTP status alone are never sufficient validation evidence. The complete source-set commitment and per-source convergence must match. Every Elasticsearch JSON response rejects duplicate object keys; every search requires `hits.total.relation == "eq"` and proves no timeout or partial shard/task failure.
6. The alias changes only through one atomic remove/add action from the exactly observed current target to the exactly validated candidate. On any precondition failure, the prior target remains the sole readable index and no success is reported.
7. After switch, repeated reads resolve exactly one candidate version and preserve retrieval, rerank, sufficiency, citation, cache, and durable evidence contracts. Historical reads retain their recorded index version and sources.
8. The predecessor, handoff watermark, and bounded rollback lease are durably recorded for CB-114 without executing a reverse switch. No physical index is deleted.
9. Elasticsearch outage, timeout, partial bulk/shard/task success, indexer restart, duplicate command, concurrent rebuild command, alias race, or crash before/after the alias action converges to an observable idempotent result without a multi-target alias or false completion.
10. Functional evidence proves `knowledge-indexer` uses only the server-owned versioned index family and exact alias operations. Local/CI Elasticsearch runs single-node with security disabled, so runtime denial evidence for admin, wildcard, and unrelated-index capabilities is an accepted deployment/hardening residual risk rather than a CB-113 gate. Exact MySQL and Support Redis denial evidence remains required: the indexer has no `commerce_db`, `cs_db`, Support Redis truth/admin, or cross-database access, and source owners do not dual-write Elasticsearch.
11. Real integration covers complete rebuild, concurrent publication and tombstone catch-up in every handoff phase, commitment mismatch, version gap/conflict, malformed/private input, partial bulk/shard/task failure, validation probe failure, alias race, restart/crash windows, predecessor/lease recording, and unchanged historical evidence.
12. No rollback execution, candidate/predecessor deletion, public authoring API, new ranking/sufficiency policy, evaluator, action/receipt, memory, handoff, web, deployment, real-provider access, or measured performance claim is introduced.

## Rejection paths

- Reject caller-selected index/alias/corpus/cutoff/version/validation values, reuse of an existing physical index, writes through the read alias, and an index outside the exact versioned family.
- Reject snapshots derived only from Elasticsearch, direct indexer database reads, incomplete source sets, aggregate-only validation, commitment mismatch, duplicate identities, version gaps/conflicts, stale tombstones, private fields, and arbitrary metadata.
- Reject a handoff that does not capture every accepted concurrent update through an explicit watermark, or that treats queue emptiness, wall-clock delay, bulk acceptance, count, or maximum version as convergence proof.
- Reject missing, ambiguous, or concurrently changed aliases; separate non-atomic remove/add calls; switching before all gates pass; and reporting success without rereading exactly one target.
- Reject timeout, HTTP-success partial shards/tasks, duplicate-key or otherwise malformed Elasticsearch responses, non-exact total-hit relations, incompatible mapping/analyzer/vector shape, shape-valid orphan/missing `SYNC_EVENT` markers, failed retrieval probes, redaction failure, or unavailable dependencies treated as validation success.
- Reject rollback execution, wildcard or physical-index deletion, rewriting historical retrieval evidence, cache or Elasticsearch authority over published truth, request-thread dual writes, direct cross-database access, broad admin grants, evaluation reachability inferred from mocks, or later-slice behavior implemented early.
- Reject capability claims based only on unit fakes without the pinned Elasticsearch runtime, real source snapshot commitment, concurrent CB-111 delivery, actual atomic alias action, and restart/crash recovery evidence.
