## Goal

Close the local dependency foundation by proving clean-environment orchestration, aggregate health and migration gates, integration entry points, controlled failure propagation, and resource-appropriate CI wiring across CB-010 through CB-013.

## In scope

- Make `make up`, `make down`, and `make test-integration` operate coherently across MySQL and migrations, both Redis instances, Elasticsearch/IK, and RocketMQ Broker/Proxy.
- Prove a clean first run using the explicit CB-010 local credential initialization path and no pre-existing volumes or implicit host state.
- Preserve non-destructive default shutdown; durable data is deleted only by a separately named explicit reset command if this slice needs one.
- Make aggregate startup depend on real health and successful one-shot migration completion, without fixed sleeps or hidden API startup side effects.
- Exercise all component permission, configuration, analyzer/vector/alias, and normal-message checks through the single integration entry point.
- Extend `make ci` only with deterministic integration checks that fit GitHub Actions resources; keep heavier full-topology evidence as an explicit local required command when necessary.
- Add README runtime commands only after their exact successful execution is captured.

## Acceptance criteria

1. From a clean environment, documented initialization followed by `make up` succeeds only after every required long-running dependency is healthy and all migration streams complete.
2. `make test-integration` executes the complete CB-010 through CB-013 probe set and propagates any component failure non-zero.
3. A repeat `make up` and migration run is idempotent and does not rotate credentials or destroy durable data.
4. `make down` removes the application topology predictably while preserving durable volumes.
5. Controlled migration failure and one controlled long-running dependency failure both make aggregate startup non-zero with the failing component visible.
6. CI and local entry points invoke the same scripts and assertions; resource-based CI omissions are explicit and are not reported as passes.
7. No API, worker, business schema, topic, or product behavior is introduced to manufacture an integration result.

## Rejection paths

- Reject an aggregate command that uses fixed sleeps, masks unhealthy or failed one-shot services, or succeeds after skipping an existing component probe.
- Reject hidden local defaults, dependence on untracked manual setup, implicit credential rotation, or destructive default shutdown.
- Reject duplicated local/CI implementations whose assertions can drift.
- Reject README commands or capability claims that were not actually run successfully.
- Reject product, order, agent, RAG, authentication, or evaluation claims based only on healthy dependencies.

## Out of scope

- API and worker containerization or service startup;
- business schemas, seed data, topics, authentication, commerce flows, agent behavior, retrieval content, or evaluation;
- production deployment, performance, resilience, or operational-readiness claims.

## Required evidence

- clean-environment initialization, `make up`, aggregate health, migration, `make test-integration`, repeat-run, and `make down` logs;
- all CB-010 through CB-013 evidence through the aggregate entry point;
- controlled migration and long-running dependency failures with non-zero exit propagation;
- exact CI inclusion/exclusion rationale and a passing real `make ci`;
- README commands matched to successful output;
- real Branch, PR, Commits, Tests, and Notes only at completion.
