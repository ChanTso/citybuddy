## Goal

Establish MySQL-authoritative reservation intent/status and one atomic Commerce Redis Lua admission decision over CB-050 projections, with owner-scoped polling and no false durable-order or premature RocketMQ claim.

## In scope

- Add commerce-owned reservation migration and application behavior for a stable server-generated reservation id, token-derived user ownership, activity association, canonical request intent, idempotency uniqueness, and `PENDING/ADMITTED/REJECTED` admission state.
- Define executable reservation request and status application contracts for CB-060's public adapter. Preserve stable replay for the same user, activity, idempotency key, and canonical intent; reject conflicting intent and body-supplied identity.
- Implement one atomic Redis Lua operation that validates the CB-050 activity id/version/window/state, enforces remaining projected quota, blocks an obvious repeated user, and writes the reservation projection, per-user marker, and deterministic admitted/rejected decision marker together.
- Persist MySQL reservation truth and provide a repeatedly callable owner-scoped internal polling port for `PENDING`, `ADMITTED`, and `REJECTED`. Redis is a hot projection and does not prove a durable order.
- Rebuild reservation projection state only from current MySQL reservation/activity truth, reject stale versions, and fail visibly when Redis cannot safely decide.
- Define configurable reservation and decision-marker TTL inputs that CB-060 can extend over the full broker transaction timeout, check interval, maximum-check window, and safety margin. Do not claim that checkback boundary is active yet.
- Extend only exact `commerce_app` DML grants for reservations and prove unrelated runtime identities remain denied.

## Acceptance criteria

1. Commerce migrations create stable reservation identity, user/activity ownership, canonical-intent idempotency uniqueness, legal admission states, and monotonic projection association without widening unrelated grants.
2. Same-user, same-activity, same-key, same-intent replay returns the existing reservation decision without another quota decrement; a conflicting intent or body identity substitution rejects deterministically.
3. One Lua execution atomically validates activity state, time window, and projection version; checks quota and the per-user marker; and writes exactly one admitted or rejected reservation decision with no partial counter, marker, or reservation state.
4. Concurrent admission against constrained quota never admits above projected allocation, and concurrent requests for one user cannot create multiple admissions. These Redis guards remain preliminary to CB-060's final database uniqueness.
5. MySQL is authoritative for reservation intent and status. Redis loss, eviction refusal, stale/missing projection, script failure, or indeterminate execution cannot produce a durable-order claim or silently mutate MySQL truth.
6. The internal polling port supports repeated owner-scoped reads, distinguishes `PENDING`, `ADMITTED`, and `REJECTED`, rejects cross-user access, and carries no false order, ledger, payment, or transaction-message result.
7. Reservation and decision-marker TTL configuration is explicit and can satisfy CB-060's broker window calculation; this slice does not claim checker or terminal-broker evidence.
8. Permission tests prove `commerce_app` has only required reservation DML, unrelated identities are denied, runtime identities cannot perform DDL, and application configuration contains no migration/bootstrap credentials.
9. No public half-message-backed route, RocketMQ producer/checker/consumer, durable seckill order, final `(activity_id, user_id)` database uniqueness, inventory ledger, delayed cancellation, payment/refund, agent, sandbox, or web behavior is introduced.

## Rejection paths

- Reject an inactive, not-yet-open, expired, missing, stale-version, or exhausted activity; duplicate-user admission; conflicting idempotency intent; body-supplied owner; and cross-user status access.
- Reject read-then-write quota admission or any partial counter, marker, reservation projection, or decision-marker update.
- Reject Redis admission, a reservation id, or `ADMITTED` status presented as proof of a durable order, inventory-ledger movement, payment eligibility, or terminal RocketMQ result.
- Reject silent fallback when Redis is unavailable, full under `noeviction`, missing required projection state, or returns an indeterminate/script error.
- Reject public CB-060 routing, half-message/checkback, durable order consumption, final database one-user-one-order claims, or delayed cancellation implemented early.
- Reject capability claims based only on mocks without real MySQL, Commerce Redis, concurrency, atomic-script, stale-version, permission-denial, idempotency, owner isolation, and controlled-failure evidence.

## Out of scope

- public reservation route adapters and the RocketMQ half-message transaction mainline;
- durable order creation/linkage, final database one-user-one-order uniqueness, and order-create inventory ledger;
- delayed unpaid cancellation and restoration;
- payment, refund, agent behavior, evaluation sandbox behavior, web UI, deployment, and measured performance.
