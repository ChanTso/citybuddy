## Goal

Establish the Python `agent-service` conversation, turn, and append-only event/evidence lifecycle over CB-020 server-owned support sessions, with ordered `cs_db` truth, direct-user owner isolation, and a stable non-streaming chat contract before bounded agent/model behavior is added.

## In scope

- Extend the frozen direct-user `POST /api/chat` family with a stable request/response contract. Validate the direct-user JWT and require `X-Session-Id`; re-read the CB-020 support session and bind every accepted request to its persisted owner.
- Persist agent-owned conversation and turn truth in `cs_db`, scoped to the opaque server-created support session and immutable user subject. Generate trace, conversation, turn, and event identities on the server; do not accept body-supplied ownership or authoritative identifiers.
- Persist accepted user input, the deterministic CB-080 response, turn outcome, and ordered append-only event/evidence records with a unique `(trace_id, sequence)` or equivalent ordering invariant. One accepted turn cannot create two logical sequences.
- Define explicit turn lifecycle and failure records so a committed accepted request has a durable terminal result or durable failure evidence. Database writes required for one accepted turn use the smallest atomic boundary that prevents a false successful response or partial ordered history.
- Make same-intent request replay deterministic through a server-defined idempotency/correlation boundary. Conflicting reuse, duplicate sequence, reordering, and concurrent requests cannot overwrite or splice another turn's evidence.
- Return only the bounded user-facing response contract. Full persisted evidence remains server-side; raw internal event payloads, private evidence, tool output, retrieval internals, credentials, and database errors are not returned.
- Keep the response implementation deterministic and local to this lifecycle slice. Do not add model calls, model routing, ToolSpecs, commerce tools, OBO acquisition, retrieval, SSE, feedback, summarization, handoff, PendingAction, or ActionReceipt behavior.
- Extend only exact `agent_app` DML grants required for the new `cs_db` conversation/event/evidence tables and prove `auth_app` and `commerce_app` remain denied.

## Acceptance criteria

1. `POST /api/chat` enforces fixed issuer, user audience, direct-user type, principal state/permission, token-derived subject, and ownership of `X-Session-Id`; missing, forged, cross-user, wrong-token-mode, and production evaluation context reject before conversation reads or writes.
2. Accepted requests use server-generated stable conversation, trace, turn, and event identities and persist the owner/session binding in `cs_db`. Request bodies cannot choose or substitute owner, session truth, trace truth, or sequence.
3. Conversation and turn records have explicit lifecycle state and stable correlation/idempotency. Same-intent replay returns the existing durable result, while conflicting reuse rejects without another logical turn.
4. Event/evidence records are append-only and strictly ordered per trace/turn through a database uniqueness invariant such as `(trace_id, sequence)`. Duplicate, reordered, and concurrent writes cannot overwrite prior evidence or cross a session boundary.
5. The required accepted-input, deterministic response/outcome, and ordered evidence writes commit or roll back in the documented `cs_db` transaction boundary. A database failure cannot return a false successful response or leave an accepted partial sequence.
6. Reading or replaying a turn always rechecks current persisted session ownership. Unknown trace, unknown session, cross-user trace/session pairing, and mismatched correlation fail closed without existence or evidence disclosure.
7. The non-streaming response exposes only the allowlisted user-facing schema. Raw evidence, database details, tool/retrieval payloads, secrets, token claims, and internal exception text remain server-side.
8. Permission tests prove only required `agent_app` DML on agent-owned `cs_db` tables, denial to `auth_app`/`commerce_app`, runtime-DDL denial, and absence of migration/bootstrap credentials in application configuration.
9. No model/provider call, agent loop, model routing, ToolSpec, OBO exchange use, commerce tool, retrieval, SSE, feedback, summary, handoff, action, evaluation evidence route, or later behavior is introduced.

## Rejection paths

- Reject wrong issuer/audience/type, missing permission, expired/not-yet-valid token, forged or cross-user support session, body identity substitution, missing session header, and production evaluation header.
- Reject client-chosen authoritative conversation/trace/turn/event identity, conflicting idempotency or correlation reuse, duplicate sequence, non-monotonic ordering, mutable evidence, and cross-session evidence association.
- Reject accepted-turn writes split so that success can be returned before required durable truth, partial evidence survives rollback, or retry creates a second logical turn.
- Reject public responses that expose raw event/evidence payloads, SQL or internal exception text, token contents, credentials, raw tool/retrieval data, or another user's existence.
- Reject Support Redis, in-memory state, request payload, logs, or optional tracing used as conversation/evidence truth in place of `cs_db`.
- Reject model, tool, OBO, retrieval, streaming, feedback, summarization, handoff, PendingAction/ActionReceipt, sandbox management, or later work implemented early.
- Reject capability claims based only on in-memory fakes without real MySQL ordering, uniqueness, ownership, rollback, replay/conflict, and permission-denial evidence.

## Out of scope

- model calls, bounded agent loops, business-tier routing, ToolSpecs, OBO exchange use, and commerce tool execution;
- retrieval/reranking, FAQ answering, Support Redis summaries, SSE, feedback, evaluation evidence APIs, and deterministic end-to-end model fakes;
- human handoff/ticket behavior, PendingAction, ActionReceipt, sandbox lifecycle, web UI, deployment, and measured performance.
