## Goal

Complete the reservation transaction-message mainline from public request through bounded Lua decision/checkback and idempotent durable seckill-order creation, while preserving one atomic MySQL order/reservation/order-create-ledger boundary and leaving delayed cancellation to CB-061.

## In scope

- Expose the frozen direct-user `POST /api/seckill/activities/{activityId}/reservations` and `GET /api/reservations/{reservationId}` routes over CB-051's application contracts. Derive ownership from the validated token, preserve request idempotency, and return explicit statuses without claiming an order before MySQL commits one.
- Send one uniquely identified RocketMQ transaction half message before invoking the CB-051 Lua admission operation. Commit only an admitted decision marker, roll back only a rejected marker, and return `UNKNOWN` only while the durable marker is absent or temporarily indeterminate.
- Implement a transaction checker that reads only the durable decision marker. Pin and expose broker transaction timeout, check interval, and maximum check count; configure marker and reservation TTLs to cover the complete checkback window plus a bounded safety margin.
- Persist one stable `transaction_resolution_due_at` with each reservation intent, derived once from the configured transaction timeout, check interval, maximum check count, and safety margin and never recomputed after restart. Run only a CB-060 transaction-decision worker that selects indexed due `PENDING` reservations in bounded batches, atomically creates a timeout decision marker only when no durable decision exists, and idempotently converges MySQL to the marker. This narrow deadline decision mechanism is not a generalized recovery scanner.
- Prove the terminal broker result when repeated `UNKNOWN` reaches the configured boundary. Unlimited checker retries or a permanently indeterminate reservation are not acceptable completion.
- Consume committed messages idempotently. In one MySQL transaction, enforce durable one-user-one-activity order uniqueness, create or return the existing owned seckill order, conditionally transition the reservation, and record exactly one order-creation inventory/activity-quota ledger movement keyed by the stable business event.
- In that same order transaction, persist the stable unpaid deadline plus timeout scheduling/event identity that CB-061 will dispatch. Do not publish or consume a delayed message in this slice.
- Add only the minimum append-only inventory-ledger capability required for order creation and replay idempotency. Define the stable movement identity and state needed by CB-061 without implementing cancellation/restoration early.
- Keep transaction messages and normal Outbox responsibilities separate. RocketMQ delivery triggers work; current MySQL state remains order and inventory truth.
- Complete `SP-003` with real Broker/Proxy evidence for rejection rollback, committed duplicate delivery, lost/unknown second-phase checkback, bounded terminal outcome, and TTL coverage.
- Extend only exact `commerce_app` DML grants required for order/reservation/order-create-ledger families and prove unrelated runtime identities remain denied.

## Acceptance criteria

1. Public reservation submission and polling enforce the CB-020 direct-user chain, token-derived ownership, idempotency, and stable schemas; body identity, cross-user access, and invalid token modes reject.
2. A Lua rejection marker rolls back the half message and produces no downstream delivery or durable order. An admitted marker commits it. A missing or temporarily indeterminate marker produces only bounded `UNKNOWN`.
3. The checker reads no database, request payload, in-memory result, or inferred Redis state other than the durable decision marker. Marker and reservation TTLs exceed the complete configured broker window and documented safety margin.
4. The configured `UNKNOWN` drill reaches an observed terminal broker result. Retries without a marker cannot continue indefinitely or be reported as a successful reservation/order.
5. Repeated or reordered committed delivery creates at most one durable order for an activity and user, transitions the owned reservation legally, and applies one order-create ledger movement. A duplicate resolves to the existing result without another decrement.
6. Order creation, reservation transition, database uniqueness, the order-create ledger movement, stable unpaid deadline, and timeout scheduling/event identity commit or roll back in one MySQL transaction. Redis cannot manufacture, repair, or override the durable result.
7. Every newly created order has exactly one durable timeout scheduling identity for CB-061; replay returns the same identity. This slice neither publishes nor consumes the delayed message.
8. Consumer database failure is not acknowledged as success, and replay after partial or ambiguous delivery remains harmless through database uniqueness, stable ledger keys, and conditional transitions.
9. Permission tests prove `commerce_app` has only new runtime DML, unrelated identities cannot access order/reservation/ledger families, runtime identities cannot perform DDL, and application configuration contains no migration/bootstrap credentials.
10. No delayed cancellation/restoration, payment/refund ledger, agent ToolSpec, sandbox, web UI, generalized recovery scanner, or later-slice behavior is introduced. The only permitted recovery mechanism is the bounded, indexed deadline decision worker scoped to this slice's reservation transaction state machine; it creates durable decision markers through Redis Lua compare-and-set and idempotently converges MySQL.

## Rejection paths

- Reject sending the half message after Lua, committing without an admitted marker, rolling back without a rejected marker, or deriving checkback from anything except the durable decision marker.
- Reject unbounded/permanent `UNKNOWN`, undocumented broker terminal settings, TTL shorter than the complete checker window, or evidence that observes only an intermediate result.
- Reject acknowledgement before the atomic MySQL order transaction commits, duplicate decrement, missing durable `(activity_id, user_id)` uniqueness, or a ledger movement without a stable replay key.
- Reject an order transaction without a stable unpaid deadline and timeout scheduling/event identity, a replay that creates a second identity, or delayed-message publication implemented early.
- Reject Redis admission presented as a durable order, Redis counters used to restore MySQL truth, payload state used instead of current MySQL state, or cross-store dual writes treated as one transaction.
- Reject an in-memory or restart-recomputed transaction deadline, an unindexed or unbounded deadline scan, interpreting Redis failure as a missing marker, overwriting an existing admitted marker, or deriving timeout from RocketMQ callback count or message payload.
- Reject delayed cancellation, inventory/activity restoration, payment/refund, or later behavior implemented early.
- Reject body owner substitution, cross-user polling, conflicting idempotency intent, invalid token mode, widened grants, or runtime DDL capability.
- Reject capability claims based only on mocks without real MySQL, Commerce Redis, RocketMQ transaction/checkback behavior, duplicate delivery, terminal-boundary, permission-denial, and controlled-failure evidence.

## Out of scope

- delayed unpaid cancellation, delay-message consumption, inventory/activity-quota restoration, and restoration ledger entries;
- mock payment, callback authentication, payment/refund state machines, payment/refund ledger movements, and reconciliation;
- agent tools, PendingAction, ActionReceipt, support evidence, evaluation sandbox behavior, and web UI;
- generalized recovery scanning, deployment, broad resilience, and measured performance.
