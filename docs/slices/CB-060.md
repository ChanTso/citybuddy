## Goal

Complete the seckill reservation mainline with RocketMQ transaction half messages, CB-050 Lua decisions, bounded checkback, idempotent durable order creation, minimum inventory-ledger movements, and delayed unpaid cancellation while preserving MySQL authority and explicit terminal outcomes.

## In scope

- Expose the frozen direct-user `POST /api/seckill/activities/{activityId}/reservations` and `GET /api/reservations/{reservationId}` routes over the CB-050 reservation application boundary. Derive ownership only from the validated token, preserve request idempotency, and return explicit reservation states without claiming a durable order before MySQL commits one.
- Send one uniquely identified RocketMQ transaction half message before invoking the CB-050 atomic Lua admission operation. Commit only an admitted decision marker, roll back only a rejected marker, and return `UNKNOWN` only when the durable marker is absent or temporarily indeterminate.
- Implement a transaction checker that reads only the durable decision marker. Pin and expose the broker transaction timeout, check interval, and maximum check count; configure marker and reservation TTLs to cover the complete checkback window plus bounded safety margin.
- Prove the terminal broker result when repeated `UNKNOWN` reaches the configured boundary. The application must not treat unlimited checker retries or a permanently indeterminate reservation as acceptable completion.
- Consume committed transaction messages idempotently. In one MySQL transaction, enforce durable one-user-one-activity order uniqueness, create or return the existing owned seckill order, conditionally transition the reservation, and record the inventory/activity-quota ledger movement keyed by the stable business event.
- Add the minimum commerce-owned inventory ledger required for seckill order creation, unpaid cancellation, inventory and activity-quota restoration, and replay idempotency. Each movement has a stable business key and cannot be applied twice.
- Publish a uniquely identified delayed message for the unpaid-order deadline. Its consumer re-reads MySQL, conditionally cancels only an eligible unpaid order, restores inventory and activity quota exactly once through the ledger, and leaves paid, cancelled, expired, or otherwise final orders unchanged.
- Keep transaction, delay, and normal Outbox event responsibilities separate. RocketMQ delivery is a trigger, not order or inventory truth; all user-visible durable results come from current MySQL state.
- Complete `SP-003` with real Broker/Proxy evidence for rejection rollback, committed duplicate delivery, lost/unknown second-phase checkback, bounded terminal outcome, and TTL coverage of the configured broker window.
- Extend only the exact `commerce_app` DML grants required by this slice and prove unrelated runtime identities remain denied.

## Acceptance criteria

1. The public reservation submission and polling contracts enforce the CB-020 direct-user identity chain, token-derived ownership, required idempotency, and stable success/rejection schemas; body identity, cross-user access, and invalid token modes reject.
2. A Lua rejection marker rolls back the half message and produces no downstream delivery or durable order. An admitted marker commits it. A missing or temporarily indeterminate marker produces only bounded `UNKNOWN`.
3. The checker reads no database, request payload, in-memory result, or inferred Redis state other than the durable decision marker. Marker and reservation TTLs exceed the complete configured broker timeout/check-interval/max-check window and its documented safety margin.
4. The configured `UNKNOWN` drill reaches an observed terminal broker result. Retries without a marker cannot continue indefinitely or be reported as a successful reservation/order.
5. Repeated or reordered committed delivery creates at most one durable order for an activity and user, transitions the owned reservation legally, and applies one order-creation ledger movement. A duplicate returns or projects the existing result without another decrement.
6. Order creation, reservation transition, and the required inventory-ledger movement commit or roll back together. Redis quota or reservation data cannot manufacture, repair, or override the durable result.
7. The delayed cancellation consumer re-reads MySQL and uses conditional state/version changes. An eligible unpaid order is cancelled once and receives exactly one inventory and activity-quota restoration; paid or final orders are never cancelled or restored.
8. Consumer database failure is not acknowledged as success, retry remains bounded by the configured broker policy, and replay after partial or ambiguous delivery remains harmless through database uniqueness, ledger keys, and conditional transitions.
9. Permission tests prove `commerce_app` has only the new runtime DML, unrelated identities cannot access order/reservation/ledger families, runtime identities cannot perform DDL, and application configuration contains no migration/bootstrap credentials.
10. No payment/refund ledger extension, agent ToolSpec, evaluation sandbox, web UI, recovery scanner, or later-slice behavior is introduced.

## Rejection paths

- Reject sending the half message after Lua, committing without an admitted marker, rolling back without a rejected marker, or deriving checkback from request memory, MySQL, message payload, or anything except the durable marker.
- Reject an unbounded or permanent `UNKNOWN`, undocumented broker terminal settings, TTL shorter than the complete checker window, or evidence that observes only an intermediate result.
- Reject acknowledgement before the MySQL order transaction commits, a duplicate that decrements twice, missing durable `(activity, user)` uniqueness, or a ledger movement without a stable replay key.
- Reject a Redis admission presented as a durable order, Redis counters used to restore MySQL truth, event-payload state used instead of current MySQL state, or cross-store dual writes treated as one transaction.
- Reject delayed cancellation that trusts the message state, cancels a paid/final order, restores inventory or quota twice, skips the inventory ledger, or hides a failed conditional transition.
- Reject request-body owner substitution, cross-user reservation polling, conflicting idempotency intent, invalid token mode, widened grants, or runtime DDL capability.
- Reject capability claims based only on mocks without real MySQL, Commerce Redis, RocketMQ transaction/checkback/delay behavior, duplicate delivery, terminal-boundary, permission-denial, and controlled-failure evidence.
