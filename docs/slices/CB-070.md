## Goal

Add an owner-scoped, idempotent mock-payment flow with separately authenticated callbacks, legal MySQL-authoritative payment/order transitions, stable payment-ledger movements, and safe integration with the CB-061 unpaid-timeout boundary, while leaving refund and full reconciliation to CB-071.

## In scope

- Expose the frozen direct-user `POST /api/orders/{orderId}/mock-payment` route. Derive ownership from the validated token, require an idempotency key, re-read current MySQL order state, and create or return one stable payment attempt for an eligible owned unpaid order.
- Persist commerce-owned payment-attempt truth with stable attempt and callback correlation identifiers, canonical request intent, user/order ownership, legal attempt states, and uniqueness that makes request replay and callback replay deterministic.
- Expose the frozen internal `POST /internal/mock-payments/callback` boundary with a separate callback credential or signature. Authenticate before reading or mutating payment/order truth; do not accept a direct-user or OBO token as callback authority.
- Apply one legal successful mock-payment transition from current MySQL truth. The payment attempt, `UNPAID → PAID` order transition, stable payment-ledger movement, and callback idempotency/audit record commit or roll back together in one MySQL transaction.
- Return the existing result for duplicate requests and callbacks with the same canonical intent. Reject conflicting idempotency or correlation reuse, unknown attempts/orders, owner mismatch, and illegal order/payment states without another ledger movement.
- Extend the CB-061 timeout consumer boundary so current paid/final MySQL state is preserved under early, duplicate, reordered, or late timeout delivery. Timeout payload state is never payment authority, and payment cannot revive an order already legally cancelled.
- Keep mock payment local and deterministic. Do not call an external payment provider or replace the authenticated callback and durable transaction with an in-memory shortcut.
- Extend only exact `commerce_app` DML grants required for payment-attempt, callback/audit, order, and payment-ledger families; prove unrelated runtime identities remain denied.

## Acceptance criteria

1. Direct-user payment submission enforces issuer/audience/type, token-derived ownership, order eligibility, canonical intent, and a user/order-scoped idempotency key; body identity and cross-user access reject.
2. One eligible request creates one stable payment attempt and callback correlation. Same-intent replay returns it, while conflicting intent or correlation reuse rejects without another attempt.
3. The callback boundary uses a separate validated credential or signature and fails closed before business reads or writes on missing, invalid, expired, or mismatched authentication.
4. A valid callback re-reads MySQL and atomically commits the legal payment-attempt transition, `UNPAID → PAID` order transition, exactly one stable-key payment-ledger movement, and callback idempotency/audit truth.
5. Duplicate and reordered callbacks return the existing result without a second state transition or ledger movement. Unknown correlation, failed/ineligible attempt, cancelled/final order, amount/order mismatch, and illegal transition reject deterministically.
6. Database failure rolls back attempt, order, ledger, and callback truth together; an ambiguous retry remains harmless through uniqueness, conditional state/version checks, and stable business-event keys.
7. CB-061 timeout delivery re-reads current MySQL truth and cannot cancel or restore inventory for a paid/final order. A late payment callback cannot revive an already cancelled order or reverse prior restoration.
8. Permission tests prove only required `commerce_app` DML, unrelated-runtime denial, runtime-DDL denial, and absence of bootstrap/migration or callback secrets in application configuration and committed files.
9. No refund state machine, refund ledger, full payment/refund reconciliation, real payment-provider integration, agent action, sandbox lifecycle, web UI, or later behavior is introduced.

## Rejection paths

- Reject wrong issuer/audience/type, missing permission, body-supplied owner, cross-user order access, missing/conflicting idempotency, unknown order, or paid/cancelled/ineligible order at payment submission.
- Reject callback authentication based on a direct-user/OBO token, caller-supplied internal identity, committed shared secret, or payload fields without the separate credential/signature boundary.
- Reject unknown or mismatched attempt/correlation/order/amount, illegal attempt state, duplicate callback that writes again, and callback idempotency reuse with conflicting intent.
- Reject payment/order/ledger/callback writes split across transactions, acknowledgement before commit, error swallowing, or a ledger movement without a stable replay key.
- Reject timeout cancellation/restoration based only on payload state, cancellation of a paid/final order, payment resurrection of a cancelled order, or double inventory/activity restoration.
- Reject Redis, RocketMQ delivery, in-memory state, or mock-provider response used as payment truth in place of current MySQL state.
- Reject refund/reconciliation, agent, evaluation sandbox, deployment, or later behavior implemented early.
- Reject capability claims based only on mocks without real MySQL transitions, authenticated-callback rejection, duplicate/reordered callback, timeout race, rollback, stable-ledger, permission-denial, and controlled-failure evidence.

## Out of scope

- refund request/processing/success/failure states, refund ledger movements, and full payment/refund reconciliation;
- external payment-provider integration, real money movement, settlement, chargeback, and provider webhook operations;
- agent PendingAction/ActionReceipt integration, evaluation sandbox behavior, web UI, deployment, and measured performance.

## Required evidence

- executable API/contract evidence for owner-scoped payment submission, same-intent replay, conflicting intent, wrong token modes, cross-user denial, order eligibility, and stable schemas;
- real MySQL evidence for payment-attempt/callback uniqueness, legal and illegal transitions, one atomic attempt/order/payment-ledger/callback transaction, rollback, and harmless retry after ambiguity;
- authenticated callback evidence for valid, missing, invalid, expired, replayed, mismatched, and conflicting credentials/signatures and correlations without secret exposure;
- real CB-061 timeout interaction evidence for early, duplicate, reordered, and late delivery preserving paid/final orders, rejecting cancelled-order resurrection, and preventing double restoration;
- migration and permission output including unrelated-runtime and runtime-DDL denial;
- evidence that no refund/reconciliation, external provider, CB-071, agent, sandbox, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
