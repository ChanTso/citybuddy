## Goal

Expose one bounded, evaluation-profile-only agent evidence view for a sandbox-associated trace, using only authoritative allowed `cs_db` evidence and independent evaluator authentication, without exposing raw private/model/tool data, reading commerce truth, implementing an evaluator, or changing support behavior.

## In scope

- Add evaluation-profile-only `GET /api/eval/evidence/{traceId}` to `agent-service`, authenticated only by the independent evaluation API credential established for the evaluation boundary.
- Require one bounded `X-Eval-Sandbox-Id` and one bounded trace identifier. Resolve the trace through persisted support-session, turn, owner, and sandbox associations in `cs_db`; never trust a caller-supplied user, session, turn, evidence category, or ownership assertion.
- Return a fixed allowlist projected from existing authoritative append-only agent evidence. The view may include bounded server-owned trace/session/turn correlation, ordered event kind/outcome/time, routing and attempt decisions, tool/retrieval decision references, terminal outcome, and public source/version identifiers only when those values are already persisted and explicitly safe.
- Preserve the existing evidence truth hierarchy. The endpoint is a read-only projection and does not copy evidence into a second truth table, reconstruct missing history from logs or tracing, re-query Elasticsearch, call a model or tool, or synthesize a success from partial records.
- Validate trace completeness and association before returning evidence. Ordered records use persisted sequence and deterministic bounds; a missing required accepted/terminal boundary, duplicate/conflicting sequence, or inconsistent trace/session/turn/sandbox association fails closed.
- Keep private user text, raw prompts and model payloads, reasoning, provider identifiers, credentials/tokens, raw tool arguments/results, live commerce data, database details, reversible PII maps, internal exception text, and unrelated trace evidence outside the response and logs.
- Make repeated reads side-effect free. They do not mutate support/evaluation truth, extend sandbox or identity TTL, reopen a terminal sandbox, update feedback, consume budgets, invoke dependencies, or create an evaluation score/result.
- Keep production behavior unchanged: the route and its evaluator credential chain are not loaded outside the evaluation profile, and production requests with evaluation headers do not gain an authentication or evidence fallback.
- Use only `agent_app` access to agent-owned `cs_db`. Do not grant or perform direct reads of `commerce_db`, the auth-owned provisioning record, Commerce Redis, or optional tracing; `auth_app` and `commerce_app` remain denied from agent evidence truth.
- Add real MySQL, profile/authentication, association, ordering/bounds, restart, redaction, and exact-permission evidence. Do not implement CB-104 asynchronous liveness, CB-105 sandbox callbacks, ServiceEval, commerce evaluation reads, web, deployment, or later work.

## Acceptance criteria

1. `GET /api/eval/evidence/{traceId}` is loaded only in the evaluation profile and requires the independent evaluation API credential. Production returns route-not-found; a direct-user JWT, service credential, sandbox header, or trace id alone is never an authentication fallback.
2. The request accepts only bounded trace and sandbox identifiers and rejects unknown fields or caller-selected owner, session, turn, event types, field lists, filters, joins, sorts, limits beyond the server maximum, or evidence-source overrides before database access.
3. The trace resolves through persisted `cs_db` trace-to-turn-to-support-session association, and that support session has the exact persisted sandbox binding. Unknown, cross-sandbox, cross-session, cross-turn, malformed, or conflicting associations fail closed without revealing whether another sandbox owns the trace.
4. The response contains only a fixed, bounded, versioned schema of allowed persisted evidence fields with deterministic sequence ordering and server-owned record/count/size limits. It never returns arbitrary stored payload blobs.
5. Accepted-input and terminal boundaries plus every returned intermediate record belong to the same trace, turn, session, owner, and sandbox. Missing required terminal truth, duplicate/conflicting sequence, partial committed history, or inconsistent lifecycle state rejects instead of returning a partial success view.
6. Retrieval, routing, budget, circuit, tool, feedback, and terminal facts are returned only when already persisted in their owning `cs_db` records and only through their explicit safe projection. The endpoint never re-runs retrieval/reranking, reconstructs evidence from public SSE, queries optional tracing, or infers an action result from token prose.
7. Responses and logs exclude user message text unless an explicit already-frozen safe field permits it, raw prompts/model responses/reasoning, provider or concrete-model details, credentials and JWTs, raw tool arguments/results, live commerce payloads, private retrieval metadata, PII maps, SQL, database errors, and evidence from another trace.
8. Repeated reads are deterministic for unchanged durable truth and perform no writes, TTL extension, budget/circuit mutation, model/tool/dependency call, sandbox transition, feedback mutation, or evaluator scoring. Restart preserves the same bounded view from `cs_db`.
9. Exact-permission evidence proves the route uses only existing required `agent_app` reads on agent-owned `cs_db`, `auth_app`/`commerce_app` remain denied, runtime DDL remains denied, and no cross-database join or direct commerce/auth-private read is introduced.
10. Real integration proves evaluation-profile success, production absence, invalid credential, unknown/cross-sandbox/cross-session trace denial, complete ordered evidence, terminal and insufficient/denied traces, deterministic replay after restart, response/log redaction, bounds, and permission denials without an external evaluator or real provider key.
11. No commerce state/audit/version implementation, asynchronous sandbox liveness, sandbox callback extension, ServiceEval, evidence scoring/judgment, evidence export, business mutation, web, deployment, or later behavior is introduced.

## Rejection paths

- Reject production-profile access; missing, invalid, wrong-chain, substituted, or broader credentials; and any user/service-token or anonymous fallback.
- Reject malformed or oversized sandbox/trace identifiers, body/query ownership assertions, caller-selected session/turn/event fields, unbounded limits, arbitrary filters/sorts, and unknown request fields.
- Reject unknown, cross-sandbox, cross-session, cross-turn, cross-owner, incomplete, duplicate-sequence, conflicting, or partially committed evidence without disclosing another trace's existence or payload.
- Reject a response assembled from logs, optional tracing, SSE replay, Support Redis, Elasticsearch re-query, model/tool re-execution, live commerce reads, or caller claims instead of authoritative persisted `cs_db` evidence.
- Reject raw stored payload blobs, user/private text outside the allowlist, prompts/reasoning/provider data, tokens/credentials, raw tool/retrieval data, PII maps, SQL/internal errors, arbitrary metadata, or another trace in responses and logs.
- Reject reads that mutate evidence, feedback, budgets/circuits, sandbox/identity TTL or lifecycle, support state, or create an evaluation result.
- Reject direct `commerce_db` or auth-private access, cross-database joins, broad grants, runtime DDL, and application use of migration/bootstrap credentials.
- Reject CB-104/CB-105 behavior, ServiceEval, commerce evaluation API changes, scoring/judgment, export, web, deployment, and later work implemented early.
- Reject capability claims based only on unit mocks without real MySQL association/ordering/restart, profile/authentication, permission, bounds, redaction, and denial evidence.

## Out of scope

- commerce-owned state/audit/version reads and sandbox lifecycle mutation;
- asynchronous consumer liveness, sandbox-bound payment callbacks, ServiceEval, evidence scoring/judgment, reviewed failure-candidate export, and optional tracing;
- changes to chat/SSE behavior, model/tool/retrieval execution, action/receipt truth, memory, handoff, web UI, deployment, real-provider access, and measured quality or performance.

## Required evidence

- executable OpenAPI/profile/credential contract tests for the evidence route, bounded request/response/error schemas, deterministic ordering, production absence, unknown-field rejection, and private-field allowlists;
- real evaluation credential and `cs_db` evidence for exact sandbox/trace/session/turn/owner association, unknown and cross-boundary concealment, complete accepted-to-terminal ordering, duplicate/conflicting/incomplete rejection, deterministic replay, and restart stability;
- real traces covering successful, insufficient/denied, tool/retrieval, and bounded failure outcomes while proving the endpoint only projects already-persisted allowed fields and performs no provider, tool, Elasticsearch, commerce, or tracing call;
- real MySQL and grant evidence proving `agent_app`-only reads, `auth_app`/`commerce_app` denial, runtime-DDL denial, no cross-database access, and no mutation from repeated reads;
- response and log redaction evidence for user/private text outside the allowlist, prompts/reasoning/providers, credentials/tokens, raw tool/retrieval payloads, PII maps, SQL/internal errors, and unrelated trace data;
- evidence that no CB-104 liveness, CB-105 callback, ServiceEval, commerce API, scoring/export, web, deployment, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
