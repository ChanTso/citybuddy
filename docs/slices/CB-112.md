## Goal

Add a bounded, version-aware Support Redis acceleration path for exact repeat FAQ questions, using a guarded query-hash mapping and a source-version-keyed public answer entry without allowing cache state to become publication, retrieval, sufficiency, citation, or historical evidence truth.

## In scope

- Add an `agent-service` first-level mapping from one server-normalized public query hash to one FAQ source id and positive source version. Populate it only after the existing retrieval/rerank/sufficiency path returns one unambiguous sufficient FAQ source; model output, caller metadata, and insufficient or mixed-source results cannot populate it.
- Add a `knowledge-indexer`-maintained second-level public answer entry keyed by FAQ source id and source version, plus the minimum owner-local current-version/tombstone metadata needed to reject a stale first-level mapping. Entries contain only the bounded published question/answer and safe public source metadata already allowed by the knowledge contract.
- Drive second-level version convergence only from the validated, ordered CB-111 FAQ publication/tombstone outcome. A newer publication makes the new version current; a newer tombstone removes answer availability and fences older versions. Equal replay is idempotent, and older or conflicting delivery cannot regress cache state.
- On a first-level hit, require exact normalized-query hash, positive version, current-version equality, and an exact second-level entry. Any missing, malformed, expired, stale, conflicting, or tombstoned component is a cache miss and may be removed; it is never converted into a sufficient answer.
- Preserve MySQL publication truth, Elasticsearch public projection truth, existing retrieval/rerank sufficiency policy, citations, and durable `cs_db` evidence. A valid cache hit projects the same bounded FAQ source id/version needed by the existing evidence contract; historical replay uses stored evidence and never re-reads Redis.
- Use bounded TTLs, key/value sizes, deterministic prefixes, and atomic compare/version logic. Query mappings expire no later than their answer/version eligibility; cache failure or eviction falls back to the existing bounded knowledge-search path without fabricating a hit or changing the answer contract.
- Keep Support Redis access limited to `agent-service` and `knowledge-indexer` responsibilities. Do not give the indexer database access, let commerce write Redis on the request/publish thread, use Commerce Redis, or create a cross-database/cache transaction.
- Prove the production knowledge synchronization path remains evaluation-unreachable under the CB-104 inventory. This cache adds no evaluation producer or sandbox carrier and therefore does not claim an asynchronous sandbox liveness guard.
- Use real Support Redis, RocketMQ-driven version changes, existing Elasticsearch retrieval, restart/eviction/outage, ordering, bounds, redaction, evidence, and access-isolation tests. Do not implement CB-113 through CB-115 rebuild/switch/rollback/cleanup behavior, FAQ authoring, retrieval-policy changes, web, deployment, or later work.

## Acceptance criteria

1. The first level stores only a bounded server-derived query hash mapped to exactly one FAQ id/version after the existing path proves one sufficient unambiguous FAQ result. Insufficient, denied, multi-source, malformed, model-selected, or caller-selected mappings are never cached.
2. The second level stores one bounded published FAQ answer under its exact source id/version and rejects draft/private/arbitrary metadata. A current-version/tombstone check prevents an old query mapping from serving an older answer after a newer publication or tombstone.
3. Newer CB-111 publication and tombstone outcomes converge cache version state monotonically. Equal replay is idempotent; older delivery and equal-version conflict cannot overwrite, resurrect, or delete newer eligible data.
4. A cache hit requires exact query-hash lookup, current source-version equality, a present valid version-keyed answer, and allowed public fields. Missing, expired, evicted, malformed, stale, conflicting, or tombstoned state becomes a miss, never a partial or ungrounded success.
5. A valid hit returns the same public answer/citation semantics and persists the same source id/version and decision facts required by durable retrieval evidence. Historical evidence reads remain stored-evidence-only and are unchanged by later cache or publication state.
6. Support Redis outage, timeout, authentication failure, eviction, restart, or malformed data cannot produce a false sufficient result or change publication truth. The request uses the existing bounded retrieval path or its existing denial behavior; it does not add an unbounded retry.
7. Keys, values, TTLs, normalization, and query/answer sizes are fixed and bounded. Raw user text, prompts, model/provider data, credentials, private content, SQL/errors, arbitrary fields, and another user's or sandbox's private state are not stored or logged.
8. Exact access evidence proves only `agent-service` and `knowledge-indexer` use their required Support Redis prefixes/commands. Commerce does not dual-write the cache, the indexer gains no MySQL access, and Commerce Redis/database/admin boundaries remain unchanged.
9. The executable asynchronous inventory remains production-only for FAQ publication/synchronization and proves no legal evaluation request creates this cache path's source event. No sandbox field, evaluation producer, or liveness-guard claim is introduced.
10. Real integration covers guarded first population, exact hit, normalization, multi-source/insufficient non-population, newer publication invalidation, tombstone fencing, stale/equal/conflicting delivery, eviction, malformed entries, Redis outage/recovery, restart, TTL/bounds, redaction, durable evidence, and access denial.
11. No CB-113 rebuild index/validation/alias switch, CB-114 rollback, CB-115 cleanup, publication authoring API, new ranking/sufficiency policy, evaluator, action/receipt, memory, handoff, web, deployment, provider access, or performance claim is introduced.

## Rejection paths

- Reject caller/model-selected cache keys, FAQ ids, versions, hit claims, TTLs, fields, or Redis commands; raw query text is never a key.
- Reject cache population from insufficient, denied, ambiguous, multi-source, private, malformed, or incomplete retrieval evidence and from a response not tied to one allowed FAQ source version.
- Reject stale query mappings, absent/wrong-version answers, tombstoned sources, older/equal-conflicting updates, resurrection, version regression, and cache state used to override publication or Elasticsearch truth.
- Reject treating Redis timeout, authentication failure, malformed content, eviction, or restart as a hit; reject unbounded retry or a cache-only denial that changes the existing retrieval contract.
- Reject cached answers that bypass citation/evidence requirements, rewrite historical evidence, expose private fields, or change ranking/sufficiency policy.
- Reject broad Redis/database grants, Commerce Redis use, commerce request-thread dual writes, direct indexer database reads, cross-database joins, runtime admin/DDL, and secret-bearing configuration or logs.
- Reject evaluation reachability inferred from mocks or hand-built events and reject CB-113 through CB-115 rebuild/switch/rollback/cleanup, authoring, evaluator, action, memory, handoff, web, deployment, provider, or performance work implemented early.
