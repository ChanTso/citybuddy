## Goal

Make every participating sandbox-bearing asynchronous consumer recheck commerce-owned sandbox liveness before applying work and converge inactive or mismatched work to one durable idempotent drop/archive disposition, without changing production event semantics, implementing callbacks, or creating evaluator truth.

## In scope

- Inventory the existing asynchronous ingress paths that can receive evaluation-scoped work under the frozen contracts and explicitly designate only those paths as participating. A path that cannot carry a sandbox identifier remains production-only and cannot infer one from payload text, logs, correlation ids, or ambient process state.
- Add one bounded, authenticated service-to-commerce liveness read for participating consumers outside `commerce-service`; commerce-local consumers use the same commerce-owned registry truth directly. The request carries exact sandbox and message correlation only, returns a minimal active/dead decision, and is evaluation-profile-only.
- Carry the immutable sandbox identifier in the authenticated/enveloped metadata of every participating evaluation message. Producers derive it from already-validated durable context; consumers require equality among envelope, owning durable record, and liveness request. Production messages omit evaluation context and cannot be upgraded by a header or body field.
- Check liveness before the first side effect. A consumer that spans a network call or database transaction rechecks or fences the same sandbox at its commit boundary so completion cannot race a late evaluation write into success.
- For a successful authoritative liveness decision that the sandbox is absent, inactive, expired, or otherwise terminal, or for deterministically invalid message context such as malformed, missing, cross-sandbox, or conflicting durable associations, perform no business/projection mutation. Converge the message to one owner-local durable `DROPPED_SANDBOX_DEAD`/equivalent disposition or archive record keyed by the existing message/event identity and sandbox. A failed or unavailable liveness query is indeterminate and is not a terminal decision.
- Make delivery retries idempotent. Repeated delivery of an applied message returns the existing applied outcome; repeated delivery of dropped work returns the existing drop/archive outcome; conflicting reuse of the same message identity with another sandbox or intent rejects and never overwrites the first decision.
- Treat liveness unavailability as unknown, not dead. Timeouts, authentication failure, malformed responses, database unavailability, or missing configuration fail closed without acknowledging an applied result and remain within existing bounded retry/dead-letter policy.
- Preserve ordering and version rules. A dropped older message cannot advance a projection watermark, a later active message cannot reopen a dead sandbox, and archive metadata cannot replace authoritative commerce or agent truth.
- Keep drop/archive records bounded and non-sensitive: message/event identity, sandbox, owning consumer, disposition reason code, source version where already public, and timestamps only. Exclude user payloads, prompts, tool results, credentials, tokens, database errors, and private business data.
- Extend only exact owner-local migration/runtime privileges needed for the disposition evidence. No consumer receives direct cross-service database access; non-commerce services check commerce truth only through the authenticated bounded API.
- Use real broker redelivery, sandbox completion races, restart, liveness outage, exact-permission, and production/evaluation profile evidence. Do not implement CB-105 mock-payment callbacks, ServiceEval, evaluator scoring, web, deployment, or later behavior.

## Acceptance criteria

1. Every asynchronous path documented as capable of carrying evaluation work either enforces the CB-104 guard or is proven unable to accept sandbox-bearing input. The inventory is executable/tested and does not silently leave a participating consumer unguarded.
2. A participating evaluation message carries one bounded immutable sandbox identifier derived from validated durable producer context. Envelope, durable resource association, and commerce liveness lookup must agree; body-only, caller-selected, missing, malformed, or substituted context rejects before mutation.
3. Cross-service liveness uses an evaluation-profile-only authenticated service endpoint with fixed request/response schemas and least-privilege caller identity. Production does not expose the route or accept evaluation metadata, and consumers never read `commerce_db` directly.
4. Active work is applied only after a successful current liveness decision and exact association check. Handlers with a commit window fence or recheck the sandbox so `ACTIVE → DEAD` racing the handler cannot commit a late evaluation mutation as success.
5. A successful authoritative liveness decision of absent, inactive, expired, or terminal, or deterministically invalid message context such as missing, malformed, cross-sandbox, or conflicting durable association, produces no business, cache, search, support, ledger, Outbox, callback, or external-effect mutation. It converges to one durable owner-local drop/archive disposition keyed by sandbox plus existing message identity. An unavailable or failed liveness query remains indeterminate under criterion 7 and cannot create that terminal disposition.
6. Duplicate active delivery returns the existing applied outcome, duplicate inactive delivery returns the existing drop/archive outcome, and conflicting message-identity reuse cannot change sandbox, intent, disposition, source version, or authoritative state.
7. Liveness timeout, authentication denial, malformed response, commerce/database outage, or unavailable configuration is not classified as sandbox death. The handler fails closed through existing bounded redelivery/dead-letter behavior and cannot acknowledge success or write a terminal false drop.
8. Ordering, source-version, tombstone, and idempotency invariants remain intact. Dropped work does not advance applied watermarks, dead sandboxes never reopen, and an archive record is evidence of disposition rather than business or sandbox truth.
9. Drop/archive schemas and logs use fixed reason codes and bounded public identifiers only. They exclude message payloads, user/private content, prompts/reasoning, provider/tool data, credentials/tokens, SQL, internal endpoints, and exception details.
10. Exact-grant evidence proves each disposition write is owner-local and append-only/idempotent, unrelated runtimes remain denied, runtime DDL is denied, and no new cross-database read or migration/bootstrap runtime path exists.
11. Real integration proves active application, completion-before-delivery drop, completion racing an in-flight handler, duplicate/redelivery convergence, conflicting identity rejection, restart recovery, liveness outage/timeout/auth failure, production rejection, bounds, redaction, and permission denial.
12. No mock-payment callback extension, ServiceEval, evaluator verdict/score, direct evaluator database access, production evaluation route, web, deployment, or later behavior is introduced.

## Rejection paths

- Reject an unlisted participating consumer, sandbox-bearing path without a guard, or a claim that a path is safe based only on code search without executable producer/consumer evidence.
- Reject absent, malformed, body-only, ambient, caller-selected, or substituted sandbox context; envelope/resource mismatch; cross-sandbox work; and production messages carrying evaluation context. These deterministic context failures may converge to the bounded terminal disposition, while a liveness transport or authority failure may not.
- Reject direct `commerce_db` access from agent/indexer/other consumers, unauthenticated liveness calls, broad service authority, production-profile liveness routes, caller-selected response fields, and unbounded lookup batches.
- Reject check-after-mutation ordering, a single stale preflight check across an unguarded commit window, late writes after sandbox death, and acknowledgement of applied work when current liveness is unknown.
- Reject treating timeout, 5xx, authentication failure, malformed response, configuration error, or database outage as a durable `DEAD` decision.
- Reject duplicate drop/archive rows, conflicting disposition overwrite, reopening terminal state, advancing an applied watermark for dropped work, or reconstructing business/evidence truth from archive metadata.
- Reject raw payloads, user/private data, prompts/reasoning, tool/provider output, credentials/tokens, SQL/internal errors, or secrets in disposition rows, responses, metrics, and logs.
- Reject broad grants, runtime DDL, cross-database joins, migration/bootstrap identities at runtime, CB-105 callback behavior, ServiceEval, evaluator scoring, web, deployment, or later work.
- Reject capability claims based only on mocks without real broker redelivery, durable owner-local disposition, commerce liveness, completion race, restart, outage, permission, profile, and redaction evidence.

## Out of scope

- synchronous chat/tool liveness already owned by CB-101 and commerce state/audit/version or agent evidence APIs owned by CB-102/CB-103;
- sandbox-bound mock-payment callback correlation and application owned by CB-105;
- new business workflows, event families, general inbox framework, global message archive, evaluator implementation, verdict/score storage, or direct evaluator database access;
- retrieval/reranking changes, knowledge publication/rebuild semantics, actions/receipts, memory, handoff, web UI, deployment, real-provider access, and measured performance.

## Required evidence

- an executable inventory tying each sandbox-capable asynchronous producer/consumer pair to its envelope field, durable association, liveness guard, commit fence, and applied/drop idempotency key;
- contract/profile/authentication tests for the bounded service liveness API, exact sandbox metadata, production absence, malformed/substituted context, fixed decisions, and stable public errors;
- real broker evidence for active application, completion-before-delivery, completion during an in-flight handler, redelivery, duplicate and conflicting identity, bounded outage retry, and restart convergence;
- owner-local database/projection evidence that inactive work causes no business/projection mutation, writes one bounded drop/archive disposition, does not advance applied ordering/version watermarks, and cannot reopen a dead sandbox;
- exact runtime grants and denial evidence for owner-local disposition writes, unrelated runtimes, runtime DDL, and cross-service database access;
- redaction evidence covering payloads, user/private text, model/tool/provider data, credentials/tokens, SQL/internal errors, and unrelated sandbox data;
- evidence that no CB-105 callback, ServiceEval, evaluator scoring, production evaluation route, web, deployment, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
