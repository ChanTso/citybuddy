## Goal

Establish commerce-owned seckill activity allocation and reservation truth plus an atomic Commerce Redis Lua admission primitive, without treating Redis admission as an order or prematurely implementing the CB-060 RocketMQ transaction-message and durable-order mainline.

## In scope

- Add commerce-owned migrations in `commerce_db` for stable seckill activities, MySQL-authoritative allocated quota, unique reservations, user ownership, idempotent reservation intent, and `PENDING/ADMITTED/REJECTED` admission state.
- Allocate or change activity quota only in a MySQL transaction that proves the allocation does not exceed authoritative inventory. Publish to Commerce Redis only as a versioned admission projection after commit.
- Define the executable reservation request and status data contracts consumed by the jointly owned CB-050/CB-060 API boundary, while keeping the public transaction-message route disabled until CB-060 can satisfy its complete half-message semantics.
- Implement a single atomic Redis Lua operation that validates the projected activity/version/window, enforces remaining quota, blocks an obvious repeated user, and writes the reservation plus deterministic admission decision marker together.
- Use server-generated reservation identifiers and derive the owner only from the validated CB-020 direct-user token. Preserve stable idempotent replay for the same user, activity, and canonical request intent without accepting body identity.
- Persist MySQL reservation truth and expose a repeatedly callable, owner-scoped internal application port for polling `PENDING`, `ADMITTED`, and `REJECTED` status. This satisfies CB-050's polling outcome at the commerce application boundary; CB-060 only exposes that established port through the frozen public `GET /api/reservations/{reservationId}` route while adding durable order states. Redis remains a hot admission projection and never proves a durable order.
- Make projection rebuild/reconciliation read current MySQL activity and reservation truth, reject stale versions, and fail visibly when Redis cannot safely admit. Do not replenish or rewrite MySQL quota from Redis counters.
- Define marker and reservation TTL configuration so CB-060 can extend it over the complete broker timeout, check-interval, and maximum-check window; this slice does not claim that transaction-check boundary is active.
- Prove least-privilege separation: `commerce_app` receives only required DML for the new families, while `auth_app` and unrelated identities cannot read or mutate seckill truth.

## Acceptance criteria

1. Commerce migrations create stable activity and reservation identities, authoritative allocation/quota fields, user ownership, idempotency uniqueness, legal admission states, and monotonic projection versions without widening unrelated grants.
2. Creating or changing an activity allocation is one MySQL transaction and rejects any allocation above authoritative inventory; a rollback leaves neither changed allocation nor a publishable projection state.
3. The Commerce Redis projection is derived only after MySQL commit, carries the activity/version/window and remaining admission quota, and cannot overwrite a newer projection with stale state.
4. One Lua execution atomically validates activity state, time window, and projection version; checks quota and the per-user marker; then records exactly one admitted or rejected reservation decision without partial counter, marker, or reservation writes.
5. Concurrent admission against constrained quota never admits above the projected allocation, and concurrent requests for one user do not create multiple admissions. These Redis guards remain preliminary to CB-060's final database uniqueness.
6. A same-user, same-activity, same-intent idempotent replay returns the existing reservation decision without another quota decrement. A conflicting intent under the same key and request-body identity substitution reject deterministically.
7. MySQL remains authoritative for activity allocation and reservation state. Redis loss, eviction refusal, stale/missing projection, script failure, or an indeterminate result cannot produce a completed-order claim or silently mutate MySQL truth.
8. The internal polling port supports repeated owner-scoped status reads, distinguishes `PENDING`, `ADMITTED`, and `REJECTED`, and carries no false durable-order result. Its executable contract and tests are completed in CB-050; only the frozen public GET adapter and `ADMITTED`-to-order transition remain CB-060 work.
9. Permission tests prove `commerce_app` has only required runtime DML on activity/reservation families, `auth_app` is denied access, runtime identities cannot perform DDL, and migration/bootstrap identities are absent from application configuration.
10. No RocketMQ transaction producer/checker/consumer, public half-message-backed reservation route, durable seckill order, final `(activity_id, user_id)` order uniqueness, inventory ledger, delayed cancellation, payment/refund, agent tool, evaluation sandbox, or web UI behavior is introduced.

## Rejection paths

- Reject activity allocation inferred from Redis, an allocation above MySQL inventory, projection publication before database commit, or a stale projection replacing a newer version.
- Reject non-atomic quota decrement, user-marker creation, reservation projection, or decision-marker writes; reject read-then-write admission that can oversubscribe under concurrency.
- Reject an inactive, not-yet-open, expired, missing, stale-version, or exhausted activity; duplicate-user admission; conflicting idempotency intent; body-supplied owner; and cross-user status access.
- Reject Redis success, a reservation identifier, or `ADMITTED` status presented as proof of a durable order, inventory-ledger movement, payment eligibility, or completed transaction-message outcome.
- Reject silent admission fallback when Redis is unavailable, full under `noeviction`, missing required projection state, or returns an indeterminate/script error.
- Reject early RocketMQ half-message/checkback behavior, durable order consumption, final database one-user-one-order claims, public CB-060 status polling, or unbounded retry/reconciliation loops.
- Reject capability claims based only on mocks without real MySQL, Commerce Redis, concurrency, atomic-script, stale-version, permission-denial, idempotency, and controlled-failure evidence.

## Out of scope

- RocketMQ transaction half messages, commit/rollback/checkback, bounded `UNKNOWN`, order consumption, and the public reservation submission mainline;
- final seckill-order uniqueness, inventory-ledger movements, durable order linkage, public reservation polling, and delayed unpaid cancellation;
- payment, refund, full reconciliation, agent ToolSpecs/OBO commerce tools, evaluation sandbox behavior, and web UI;
- production deployment, broad resilience, and measured performance or operational-readiness claims.

## Required evidence

- migration and permission output for activity and reservation ownership, including explicit `auth_app` denial and runtime DDL denial;
- real MySQL transaction evidence for legal allocation, over-allocation rejection, rollback, monotonic projection version, and no Redis-to-MySQL truth reversal;
- real Commerce Redis evidence for projection publication after commit, stale-version rejection, activity-window validation, noeviction failure, and MySQL-based rebuild;
- atomic Lua evidence for admitted, quota-exhausted, inactive, duplicate-user, stale/missing projection, same-intent replay, conflicting-intent, and script/connection failure paths;
- constrained concurrent-user and same-user tests proving projected quota is not oversubscribed and one user's replay does not decrement twice;
- repeated owner-scoped internal polling evidence, including cross-user rejection and state changes, proving no response or projection claims a durable order and no public GET or CB-060 RocketMQ/order behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
