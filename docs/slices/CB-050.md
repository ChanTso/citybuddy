## Goal

Establish commerce-owned seckill activity allocation in MySQL and its versioned Commerce Redis admission projection, without introducing reservation, Lua admission, RocketMQ, or durable seckill-order behavior early.

## In scope

- Add commerce-owned migration and application behavior for stable seckill activities, an activity window and state, MySQL-authoritative allocated quota, and a monotonically increasing projection version.
- Allocate or change activity quota only in a MySQL transaction that proves the allocation does not exceed authoritative inventory and does not violate the current activity state.
- Publish the Commerce Redis admission projection only after MySQL commit. Include the stable activity id, projection version, activity window/state, and remaining projected quota needed by CB-051.
- Reject a stale projection write so an older activity version cannot overwrite a newer Redis value.
- Rebuild or reconcile the projection only from current MySQL activity truth. Redis loss, eviction refusal, or corruption never replenishes or rewrites MySQL allocation.
- Fail closed when the versioned projection cannot be written safely under the Commerce Redis `noeviction` policy.
- Extend only the exact `commerce_app` DML grants required for the activity family and prove unrelated runtime identities remain denied.

## Acceptance criteria

1. Commerce migrations create a stable activity identity, authoritative allocated-quota fields, legal activity window/state constraints, and a monotonic projection version without widening unrelated grants.
2. Creating or changing an activity allocation is one MySQL transaction and rejects any allocation above authoritative inventory; rollback leaves neither a changed allocation nor a publishable projection state.
3. The Redis projection is derived only from committed MySQL state and carries the activity id, version, window/state, and remaining projected quota required by CB-051.
4. A stale or reordered projection write cannot overwrite a newer version.
5. Projection rebuild and reconciliation read MySQL truth, recreate missing state, and never derive MySQL inventory or allocation from Redis counters.
6. Redis unavailable, full under `noeviction`, malformed, or indeterminate projection writes fail visibly and cannot be reported as an admitted reservation, order, or durable inventory result.
7. Permission tests prove `commerce_app` has only required runtime DML for the activity family, `auth_app` and unrelated identities are denied, runtime identities cannot perform DDL, and application configuration contains no migration/bootstrap credentials.
8. No reservation identity/state, per-user marker, Lua admission, polling, RocketMQ transaction producer/checker/consumer, durable seckill order, inventory ledger, delayed cancellation, payment/refund, agent tool, sandbox, or web UI behavior is introduced.

## Rejection paths

- Reject activity allocation inferred from Redis, allocation above MySQL inventory, invalid activity windows/states, and rollback that leaves a publishable projection.
- Reject projection publication before database commit, a stale version replacing a newer value, or cross-store writes described as one transaction.
- Reject Redis loss, noeviction refusal, or malformed projection data that silently falls back to admission or mutates MySQL truth.
- Reject reservation/Lua/user-marker/decision-marker behavior, public reservation routes, RocketMQ ordering, ledger movement, or later-slice behavior implemented early.
- Reject capability claims based only on mocks without real MySQL, Commerce Redis, stale/reordered writes, permission denial, rebuild, rollback, and controlled-failure evidence.

## Out of scope

- reservation request/status truth, owner-scoped polling, idempotent reservation intent, and atomic Lua admission;
- RocketMQ transaction half messages, commit/rollback/checkback, order consumption, final order uniqueness, and inventory ledger;
- delayed unpaid cancellation and restoration;
- payment, refund, agent behavior, evaluation sandbox behavior, web UI, deployment, and measured performance.

## Required evidence

- migration and permission output for activity ownership, including explicit unrelated-runtime denial and runtime DDL denial;
- real MySQL evidence for legal allocation, over-allocation rejection, rollback, legal window/state validation, and monotonic projection version;
- real Commerce Redis evidence for publication after commit, stale/reordered version rejection, missing-projection rebuild, `noeviction` failure, and no Redis-to-MySQL truth reversal;
- controlled-failure evidence proving no projection error produces reservation, order, or inventory-success claims;
- evidence that no CB-051 or later behavior was introduced;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
