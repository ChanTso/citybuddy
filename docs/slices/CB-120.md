## Goal

Add the commerce-owned sensitive-action boundary that prepares an owner- and turn-bound PendingAction,
then confirms it by atomically validating and consuming that action, executing one legal business
mutation, and persisting one immutable ActionReceipt. Retries converge on durable MySQL truth and
cannot execute the business mutation twice.

## In scope

- Add closed-schema internal prepare and confirm endpoints under the existing exact-scope Agent OBO
  boundary. Identity, owner, actor, support session, trace, turn, scope, and evaluation sandbox are
  derived from validated credentials and server context, never from model-controlled body fields.
- Introduce commerce-owned PendingAction and ActionReceipt migrations, repositories, and exact runtime
  grants. PendingAction records the server-derived action id, action type, canonical argument
  commitment, owner, support session, trace/turn correlation, target resource identity and version,
  expiry, state/version, and one server-derived idempotency key. ActionReceipt records the immutable
  authoritative result, action identity, canonical intent commitment, resulting resource identity and
  version, owner and correlation, and one stable receipt idempotency key.
- Keep the executable action registry closed and server-owned. CB-120 integrates the existing
  owner-scoped refund-request business boundary as the initial sensitive mutation; callers cannot
  select a class, method, URL, SQL fragment, scope, target owner, resource version, receipt shape, or
  arbitrary action implementation.
- On prepare, validate the exact action schema, authorization, ownership, resource eligibility and
  current version, then persist or replay one PendingAction derived from
  `(owner, support session, turn, action type, canonical arguments)`. Same-key/same-intent replay
  returns the existing action; any commitment mismatch rejects.
- On confirm, lock and re-read the PendingAction and relevant business truth. In one
  `commerce_db` transaction validate owner, session, action type, scope, canonical argument hash,
  resource identity/version, expiry, sandbox binding/liveness when present, and unconsumed state;
  execute the legal refund request through the owning commerce transaction boundary; consume the
  PendingAction once; and persist the immutable ActionReceipt. Any failure rolls back the consume,
  mutation, ledger/Outbox writes, and receipt together.
- Treat a durable ActionReceipt as the only confirmation success truth. Same-intent replay returns the
  existing receipt without another refund, ledger movement, or Outbox row. Ambiguous response loss,
  duplicate requests, and bounded recognized database races re-read the receipt and business truth
  before classifying an outcome.
- Return only a bounded receipt view needed by CB-121. Do not persist agent projection or turn commit,
  emit public SSE, generate confirmation prose, or invoke a model in this slice.
- Preserve the existing direct-user refund API and CB-071 payment/refund invariants. PendingAction is
  an authorization and one-time execution boundary, not an alternative refund state machine or truth
  owner.
- Prove exact `commerce_app` access to the new table family and denial for `auth_app`, `agent_app`,
  migration-only, cross-database, and unrelated business-table capabilities.

## Acceptance criteria

1. Prepare accepts only an authenticated Agent OBO with the fixed issuer, OBO purpose/type,
   `aud=commerce-service`, `act.azp=agent-service`, exact action scope, user subject, verified support
   session, time bounds, and exact sandbox header/claim equality where applicable. Direct-user,
   evaluator, callback, wrong-service, wrong-scope, forged-session, body-identity, cross-user, and
   inactive/indeterminate-sandbox requests fail closed.
2. The action registry and request schemas are closed. The initial action is the existing
   owner-scoped refund request with server-resolved order/payment ownership and version; unknown
   action names, unknown fields, non-canonical values, caller-selected owners/versions/scopes, and
   unsupported mutations reject before persistence.
3. One canonical prepare intent produces one PendingAction with a bounded expiry and unique
   server-derived idempotency key. Same-intent replay returns it; the same key with a changed action,
   arguments, owner, session, turn, target, version, or sandbox rejects without another row.
4. Confirm revalidates every authorization and intent binding against current truth inside the
   transaction. Expired, consumed, mismatched, cross-owner/session/sandbox, stale-version,
   ineligible-resource, illegal-transition, or malformed PendingAction truth rejects without a
   consume, business mutation, receipt, ledger movement, or Outbox row.
5. Successful confirmation locks the PendingAction and affected refund/payment/order truth, consumes
   the action once, performs the legal CB-071 refund request, writes its required ledger/Outbox truth,
   and persists one immutable ActionReceipt in one MySQL transaction. Controlled failure at every
   write boundary rolls the complete unit back.
6. The ActionReceipt contains a stable receipt identity and complete immutable canonical commitment
   to the prepared action and authoritative result. Every public or cursor/order/time-bearing
   receipt column is anchored to business truth; database-generated columns have documented and
   enforced invariants. No receipt column is excluded from integrity disposition.
7. Concurrent identical confirms and replay after response loss converge to one consumed action, one
   refund result, one permitted ledger movement/Outbox effect, and one receipt. Every successful
   caller receives that same durable result; a conflicting intent is a deterministic conflict.
8. Lock timeout, uniqueness race, and recognized concurrency exhaustion re-read committed
   PendingAction, receipt, and refund truth. Failure to observe is not absence: indeterminate
   observations remain bounded/retryable and cannot be classified as conflict, denial, not found,
   success, or dependency unavailability. True database unavailability remains explicitly
   unavailable.
9. Receipt replay validates the complete canonical intent/result commitment rather than key
   membership alone. Missing, duplicate, contradictory, or partially persisted action/receipt/refund
   faces fail closed and never trigger a second mutation.
10. MySQL remains the sole action truth. Redis, RocketMQ delivery state, agent memory, request bodies,
    model text, and in-memory state cannot authorize confirmation or establish success.
11. Exact migrations and grants give only `commerce_app` the required DML. `agent-service` reaches the
    boundary only through OBO-authenticated HTTP and cannot access `commerce_db`; `auth_app` cannot
    read or mutate PendingAction, ActionReceipt, refund, ledger, or Outbox truth.
12. No agent-side confirmation interpretation, receipt projection, turn commit, public
    `action_receipt` event, model/network retry behavior, handoff, memory, web, deployment, provider,
    evaluation API, generalized recovery scanner, or later-slice behavior is introduced.

## Rejection paths

- Reject wrong issuer/audience/type/purpose/actor/scope, direct-user or evaluator credentials,
  missing/forged/cross-user support sessions, caller-supplied identity, sandbox mismatch or
  non-ACTIVE/indeterminate sandbox truth, and any authorization failure rendered as success or
  absence.
- Reject unknown action names or fields, non-canonical or oversized arguments, caller-selected
  implementation details, arbitrary URLs/SQL/scopes/owners/versions/receipt fields, and prepare
  intent that does not match current owned business truth.
- Reject idempotency comparison that omits any content-bearing action or Outbox column, filters
  contradictory rows out of its enumeration, treats an observation failure as confirmed absence, or
  accepts same-key/different-intent replay.
- Reject confirm outside one transaction, pre-consume followed by failed mutation, mutation without
  receipt, receipt without mutation, split refund/ledger/Outbox writes, acknowledgement before commit,
  or broad catch-and-continue handling after ambiguous persistence.
- Reject expired, consumed, stale-version, wrong-owner/session/turn/sandbox, altered-argument,
  ineligible-resource, over-refund, unpaid/cancelled, illegal-state, missing, duplicate, or
  contradictory durable truth.
- Reject a concurrency path that leaks raw lock/constraint failures, returns unavailable after a
  sibling committed, reports conflict without a successful observation, or executes the refund more
  than once.
- Reject mutable receipts, caller-authored success fields, deriving action success from model prose,
  agent projection, Redis, an Outbox delivery, or an HTTP status without durable receipt truth.
- Reject broad runtime grants, direct `agent_app` database access, cross-database joins, new
  asynchronous carriers, rollback/cleanup behavior, or CB-121 and later work implemented early.

## Out of scope

- agent-side confirmation-text interpretation, persistence of the PendingAction reference, receipt
  projection into `cs_db`, turn commit, SSE `action_receipt`, and post-commit explanation retry owned
  by CB-121;
- new refund state-machine or provider behavior, chargebacks, settlement, direct-user refund changes,
  generalized arbitrary sensitive actions, cancellation redesign, or a workflow engine;
- human handoff/tickets, memory/summary packing, prompt/PII/output-safety changes, web UI, deployment,
  real-provider access, observability expansion, and measured performance;
- evaluation reset/state/audit changes or a new evaluation-reachable asynchronous carrier;
- background expiry deletion or generalized recovery scanning. Expired actions remain unusable and
  may be retained as bounded durable audit truth until separately authorized lifecycle work.

## Required evidence

- executable closed-schema and canonical-commitment tests for PendingAction and ActionReceipt,
  including every content-bearing column, unknown-field rejection, stable serialization, same-intent
  replay, conflicting intent, and immutable receipt projection;
- real OBO integration for exact issuer/type/audience/actor/scope, owner/session/turn correlation,
  wrong token modes, cross-user/session denial, forged body identity, sandbox equality/liveness, and
  unchanged production-token behavior;
- real MySQL evidence for prepare uniqueness, bounded expiry, current resource/version anchoring,
  confirm-time revalidation, one atomic consume/refund/ledger/Outbox/receipt boundary, controlled
  rollback at every write point, and exact database constraints;
- deterministic concurrent confirm and response-loss evidence proving one refund, one legal movement,
  one Outbox effect, one immutable receipt, and identical same-intent results; separate evidence for
  conflicting intent, indeterminate observation, lock timeout, uniqueness race, and true database
  unavailability classification;
- data-consistency fault injection across the PendingAction, ActionReceipt, refund, order/payment,
  ledger, and Outbox faces for missing, duplicate, and per-column contradictory truth, proving
  fail-closed behavior without a second mutation;
- exact runtime grant evidence for allowed `commerce_app` operations and denial of `auth_app`,
  `agent_app`, cross-database, DDL/admin, and unrelated-table access;
- regressions proving the existing direct-user refund lifecycle, reconciliation, OBO boundary,
  evaluation isolation, support turns/SSE, and ActionReceipt-only public status contract remain
  unchanged;
- evidence that no CB-121 agent projection/turn/SSE behavior, new asynchronous carrier, generalized
  action engine, handoff, memory, web, deployment, provider, performance, or later behavior was
  implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
