## Goal

Turn the bounded CB-090 hybrid candidate set into one grounded retrieval decision by applying a role-alias-only reranker, a versioned development-set-calibrated sufficiency gate, and durable `cs_db` evidence bound to the exact turn, index version, and public sources actually used.

## In scope

- Consume only the bounded, schema-validated, deterministic-RRF candidate set produced by CB-090. The original query, optional rewrite, resolved physical index version, candidate identities, ranks, and allowlisted public text remain distinct inputs; no caller or model may replace index, source, score, rank, threshold, or evidence authority.
- Add one server-owned reranker role alias behind the existing CB-081 model policy. It receives a bounded candidate view, stays within the selected business tier and shared turn attempt budget, has finite timeout/retry behavior, and cannot select tools, scopes, providers, concrete model identifiers, or cross-tier fallback.
- Normalize reranker output against the input candidate allowlist, reject unknown/duplicate/missing identities and malformed or non-finite scores, apply deterministic tie-breaking, and bound the final ordered evidence set. Reranker prose, hidden reasoning, provider payloads, and candidates not produced by CB-090 never become evidence.
- Commit one versioned sufficiency-calibration artifact derived from a bounded public/synthetic development fixture. The server owns the score threshold and top-result margin; runtime requests and model output cannot change them. A deterministic gate returns either sufficient evidence or a structured denial/clarification outcome.
- Persist the retrieval decision in `cs_db` with the accepted turn: trace, turn, session association, resolved index version, calibration version, selected public source/chunk identities and versions, bounded rank/score metadata, and the sufficiency outcome. Persist only the evidence actually used; never re-query Elasticsearch to reconstruct or rewrite history.
- Keep terminal turn outcome and its retrieval evidence atomic at the established agent completion boundary. Same-intent replay returns the stored decision/evidence and cannot rerun retrieval or reranking; failure before commit leaves no partial retrieval evidence, and failure after an accepted turn follows the existing bounded terminal-failure convergence.
- Expose citations or source references only from the stored sufficient evidence allowlist. Insufficient, unavailable, malformed, stale-association, or uncommitted evidence cannot support an answer claim and produces a bounded denial or clarification path.
- Extend only exact `agent_app` migration/runtime privileges needed for append-only retrieval evidence and prove unrelated runtimes remain denied. Use deterministic reranker fixtures for CI; no real provider credential or measured production-quality claim is introduced.
- Keep incremental FAQ/product synchronization, tombstones, rebuild/alias switching, retrieval cache, evaluation APIs, memory summaries, sensitive actions/receipts, handoff, web, deployment, and later behavior out of this slice.

## Acceptance criteria

1. The reranker accepts only a bounded CB-090 candidate schema and uses one server-owned role alias through the CB-081 model-policy and attempt-budget boundary. No concrete provider model, caller-selected tier, proxy business routing, cross-tier fallback, or independent retry budget exists.
2. Reranker requests contain only the bounded query and allowlisted public candidate view. Raw vectors, embeddings, Elasticsearch payloads, private metadata, credentials, user/session authority, commerce live truth, and unrelated support evidence never enter or leave the reranker boundary.
3. Output validation rejects unknown, duplicate, omitted-required, malformed, non-finite, or out-of-bound candidate scores before grounding. Stable candidate identity and deterministic tie-breaking produce the same bounded order for repeated input.
4. A versioned calibration artifact records the bounded development fixture, selected score threshold, top-result margin, and deterministic derivation/check command. Runtime requests and model output cannot widen or override either gate parameter.
5. Sufficient fixtures pass and insufficient, ambiguous-margin, empty, malformed, timeout, unavailable, and exhausted-budget cases return a stable structured denial or clarification without an unsupported answer or citation.
6. Durable retrieval evidence is associated with one persisted trace, turn, session, and owner and records the exact resolved `knowledge_docs_vN` version, calibration version, sufficiency outcome, and allowlisted source/chunk identities and versions actually used.
7. Retrieval evidence and terminal turn completion commit atomically. Controlled failure after at least one attempted evidence write rolls back all partial retrieval evidence and terminal agent events; replay reads the stored outcome without another Elasticsearch or reranker call.
8. Citations and grounding claims can reference only stored sufficient evidence. Unknown/cross-session/cross-user association, a candidate not in the CB-090 result, stale or mismatched index version, and reconstruction by re-query all fail closed.
9. Real MySQL evidence proves append-only identity/association/ordering constraints, duplicate/conflicting replay rejection, rollback, exact `agent_app` DML, unrelated-runtime denial, and runtime-DDL denial.
10. Deterministic end-to-end evidence through the CB-090 real index proves hybrid retrieval, rerank normalization, sufficient and insufficient decisions, exact stored source/index binding, replay, provider denial/exhaustion, redaction, and controlled failure without a real provider key.
11. No incremental publication/tombstone consumer, index rebuild or alias switch, retrieval cache, evaluation, memory, sensitive action/receipt, handoff, web, deployment, real-provider quality claim, or later behavior is introduced.

## Rejection paths

- Reject reranking arbitrary caller/model candidates, caller-selected index/tier/provider/threshold/margin/count, concrete model identifiers, proxy-side business routing, cross-tier fallback, and an independent or resettable retry budget.
- Reject raw vectors, embeddings, Elasticsearch requests/responses, private or transactional metadata, live price/stock/payment/refund claims, credentials, prompts, provider payloads, reranker prose/reasoning, or database details at the reranker, evidence, citation, response, or log boundary.
- Reject unknown/duplicate/missing candidate identities, score NaN/infinity, oversized candidate/result lists, nondeterministic ties, candidates not emitted by CB-090, and partial rerank success after timeout, schema failure, or budget exhaustion.
- Reject unversioned or hand-tuned runtime thresholds, a calibration fixture containing private data, a threshold/margin override from a request or model, and capability claims that report fixture classification as measured production quality.
- Reject an answer or citation when the gate is insufficient, evidence is missing/uncommitted/mismatched, source or index version differs from the retrieval decision, or evidence is reconstructed by a later Elasticsearch query.
- Reject partial retrieval-evidence history, cross-owner/session/turn association, mutable overwrite, duplicate conflicting evidence, runtime DDL, or access by `auth_app`/`commerce_app`.
- Reject incremental sync, tombstones, rebuild/alias operations, Support Redis caching, evaluation, memory, actions/receipts, handoff, web, deployment, and later work implemented early.
- Reject evidence based only on unit mocks without the real CB-090 index/alias, deterministic reranker path, calibrated sufficient/insufficient fixtures, real `cs_db` atomicity/permissions, replay, rejection, and redaction checks.

## Out of scope

- FAQ/product publication truth, incremental RocketMQ indexing, source-version/tombstone mutation, two-level retrieval cache, rebuild validation, alias rollover/rollback, and cleanup authorization;
- live commerce reads/writes, OBO acquisition, sensitive PendingAction/ActionReceipt execution, memory/summary packing, evaluation identity/sandbox/evidence APIs, handoff/tickets, web UI, deployment, and measured real-provider retrieval quality or latency;
- training or fine-tuning a reranker, introducing a second agent, adding a decomposer/intent classifier, or changing CB-081 business-tier ownership.

## Required evidence

- executable reranker input/output and role-alias contract tests proving bounded candidates, no concrete model identifiers, stable normalization/ties, malformed/unknown/duplicate/non-finite rejection, shared budget, finite retry, and no cross-tier fallback;
- a committed versioned public/synthetic calibration fixture plus deterministic command/output proving threshold and top-result-margin selection, sufficient/insufficient/ambiguous classifications, and no runtime override;
- real CB-090 Elasticsearch evidence proving only fused public candidates reach reranking and the exact resolved index/source versions flow into the decision without raw vectors or live commerce authority;
- real `cs_db` migration, atomic commit/rollback, append-only association, replay, conflict, exact-grant, unrelated-runtime, and runtime-DDL-denial evidence for retrieval records bound to trace/turn/session;
- deterministic fake-provider end-to-end evidence for rerank success, denial, timeout/unavailability, malformed output, attempt exhaustion, stored sufficient/insufficient outcome, citation allowlist, replay without re-execution, and public/log redaction;
- evidence that no incremental sync, tombstone, rebuild/alias, cache, evaluation, memory, sensitive action/receipt, handoff, web, deployment, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
