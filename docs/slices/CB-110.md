## Goal

Establish commerce-owned FAQ draft and publication truth with stable identifiers, monotonically increasing published versions, and one normal-message transaction Outbox record committed atomically with every publish, without implementing withdrawal behavior, the knowledge-indexer consumer, FAQ caches, rebuild/alias changes, or a public FAQ-authoring API.

## In scope

- Add commerce-owned FAQ migrations in `commerce_db` using the existing commerce migration and runtime identities. Persist one stable FAQ identifier, bounded public question/answer content, explicit draft/publication state, and monotonically increasing published version.
- Keep draft content and current publication truth authoritative in MySQL. The existing bootstrap FAQ corpus remains development/search fixture data and is not silently promoted into commerce publication truth.
- Provide one bounded commerce-internal publication command boundary, following the existing product-publication service pattern. This slice does not add a public authoring or administration HTTP route, infer an editor identity, or create a second management credential.
- Make a publish transition and its uniquely identified, versioned FAQ Outbox event one local MySQL transaction. A rollback leaves neither changed FAQ publication truth nor an orphan event.
- Emit only the normal knowledge-synchronization envelope fixed by the asynchronous contract: event id, source id/type/version, publication state, tombstone, occurred time, and bounded public content or reference. Do not place draft-only content, user/private data, credentials, internal metadata, SQL, or errors in the payload.
- Preserve immutable event meaning and monotonic source versions. Same-intent replay returns the one durable result; reuse of an event/idempotency identity for different FAQ content, source, state, tombstone, or expected version rejects without overwriting truth.
- Reuse the existing bounded Commerce Outbox publisher semantics. Publishing progress is delivery evidence only; it cannot change FAQ business state, fabricate a newer version, or mark delivery before the broker accepts the message.
- Keep request/service code from independently dual-writing MySQL, RocketMQ, Elasticsearch, or Support Redis. This slice may publish the FAQ source event boundary but does not consume it or mutate knowledge projections.
- Extend only exact owner-local migration/runtime privileges needed for FAQ and its existing Outbox family. `auth_app`, `agent_app`, and `knowledge-indexer` remain denied direct FAQ-table access; runtime DDL and cross-database access remain denied.
- Add real MySQL, transaction rollback, normal-message publication, restart, idempotency/conflict, bounds, redaction, and exact-permission evidence. Do not implement CB-111 incremental indexing/cache behavior, CB-112 rebuild behavior, agent retrieval changes, web, deployment, or later work.

## Acceptance criteria

1. Commerce migrations create FAQ source truth with a stable bounded identifier, bounded public question/answer fields, explicit draft/publication state, database constraints, and a monotonically increasing published version; no second source of FAQ truth is introduced.
2. Draft creation or editing does not claim a published version, emit an indexable success event, appear in existing public retrieval as newly authoritative content, or write Elasticsearch/Support Redis.
3. Publishing a valid draft atomically persists the published content/version and one uniquely identified `PENDING` FAQ Outbox event. Controlled failure after either write rolls back both.
4. Every later successful publication of a revised draft for the same FAQ advances the source version exactly once. An older expected version, stale command, or out-of-order intent cannot overwrite newer durable truth.
5. Same-intent sequential and concurrent replay converges to one FAQ result, one source version, and one Outbox event. Conflicting reuse of an event or idempotency identity rejects without changing the committed FAQ or Outbox record.
6. The Outbox payload is a fixed bounded knowledge-synchronization envelope containing only event/source identifiers, `faq` source type, source version, publication state, tombstone, occurred time, and the allowed public content/reference. Draft-only or private fields and arbitrary metadata are absent.
7. Every CB-110 FAQ publication event explicitly carries published state and `tombstone=false`. The envelope retains the frozen tombstone field for CB-111 compatibility, but this slice does not claim or implement a withdrawal/delete transition.
8. The existing bounded Outbox publisher sends the FAQ event as a normal RocketMQ message and marks only delivery progress after broker acceptance. Publisher failure/restart leaves the event pending and retryable; duplicate publication cannot mutate FAQ truth.
9. No request or command path dual-writes Elasticsearch or Support Redis, and no knowledge-indexer consumer, source-version projection, FAQ answer cache, rebuild validation, or alias switch is implemented early.
10. Exact grants prove `commerce_app` has only required FAQ DML plus existing Outbox access, migration ownership remains separate, unrelated runtimes cannot read or mutate FAQ truth, runtime DDL is denied, and no cross-database path exists.
11. Real integration proves draft isolation, first publish, revised-draft republish, transaction rollback, same-intent and conflicting replay, stale expected-version rejection, bounded payload/redaction, explicit non-tombstone publication, publisher failure/retry, restart convergence, normal-message delivery, exact permissions, and absence of early consumer/cache/index writes.

## Rejection paths

- Reject malformed or oversized FAQ identifiers/content, blank public question/answer, unknown states/fields, arbitrary payload metadata, caller-selected source type, nonpositive versions, and unbounded publication commands.
- Reject draft content treated as published truth, a publication version advanced without a committed publish, in-place reuse of an old published version, or unapproved withdrawal/delete behavior.
- Reject a FAQ mutation committed without its Outbox row, an Outbox row committed for a rolled-back mutation, a false success after transaction failure, or delivery state treated as business truth.
- Reject stale expected versions, duplicate rows/events, conflicting idempotency reuse, event identities rebound to another FAQ or payload, and replay that increments the version twice.
- Reject Outbox payloads containing draft-only data, private/user content, credentials/tokens, internal account or host data, SQL/errors, arbitrary tables/fields, or an unbounded full record.
- Reject request-thread or publication-service writes to Elasticsearch/Support Redis, direct knowledge-indexer reads from `commerce_db`, or independent MySQL/broker/search/cache writes whose partial success can diverge.
- Reject broad grants, runtime DDL, auth/agent/indexer access to FAQ tables, migration/bootstrap identities at runtime, and cross-database joins.
- Reject capability claims based only on mocks without real MySQL atomicity, normal RocketMQ delivery/retry, restart, concurrent idempotency/conflict, version/tombstone, permission, bounds, and redaction evidence.
- Reject CB-111 consumer/cache, CB-112 rebuild/alias, agent retrieval changes, public authoring API, web, deployment, or later work implemented early.

## Out of scope

- public or web FAQ authoring/administration workflows, editor roles, approval UI, localization workflow, content moderation, and FAQ withdrawal/delete source transitions;
- `knowledge-indexer` consumption, Elasticsearch upsert/delete, source-version watermarks, FAQ two-level cache, tombstone application, and incremental retry semantics owned by CB-111;
- full rebuild validation, rebuild handoff, physical index replacement, and atomic alias switching owned by CB-112;
- changes to existing retrieval/reranking truth, support behavior, actions/receipts, evaluation, memory, handoff, deployment, real-provider access, and measured performance.
