## Goal

Extend the CB-080 durable support turn with one bounded ReAct agent, deterministic routing signals, business-tier model policy, executable ToolSpec mediation, and a shared attempt budget across model, proxy, HTTP, and tool work, using only role aliases and a deterministic fake-provider path for repeatable evidence.

## In scope

- Replace the deterministic CB-080 response step with one single-Agent ReAct loop while preserving the established direct-user, support-session ownership, idempotency, ordered-event, and durable-terminal-turn boundaries.
- Keep `RuleRouter` deterministic and limited to signals such as risk, private action, public FAQ eligibility, chitchat, and complexity. `ModelRouter` alone selects the business tier, escalation policy, and remaining budget; neither LiteLLM nor the model selects a business tier.
- Propagate one finite `attempt_budget` across application model attempts, LiteLLM transient/network retry and same-tier fallback, HTTP calls, and tool attempts. Every charged attempt is observable in server-side evidence, exhaustion terminates deterministically, and no layer starts an unbounded or hidden retry loop.
- Add provider-scoped circuit-breaker state with a configured minimum request count, bounded open interval, and bounded half-open probes. One provider's failures cannot open another provider's circuit or silently widen the selected tier.
- Define executable server-owned ToolSpecs for the tools introduced in this slice. Each ToolSpec fixes its input/output schema, risk tier, exact OBO scope, timeout, idempotency behavior, and bounded model-visible result; model output and request fields cannot add tools, expand scope, extend timeout, or change risk.
- Acquire the CB-020 OBO token just in time on the first commerce tool use, using the already-validated direct-user and persisted support-session binding plus the exact ToolSpec scope. Cache, if used, is keyed by user, session, and exact scope and never outlives the token.
- Route commerce tool calls only through the frozen `POST /internal/tools/{toolName}` family. `ToolAdapter` validates model arguments before I/O, sends no body-supplied identity authority, enforces timeout and response schema, exposes only the ToolSpec-bounded model view, and converts expected policy/schema/ownership/timeout denials into structured `deny_with_feedback` results.
- Persist ordered model, routing, budget, circuit, tool lifecycle, denial, and terminal outcome evidence through the CB-080 event boundary. Public non-streaming responses remain bounded and do not expose raw prompts, provider payloads, tokens, ToolSpecs, tool results, circuit internals, or database details.
- Add a local LiteLLM-compatible role-alias path backed by a deterministic fake provider for integration and CI evidence. No real provider key is present in CI, committed configuration, examples, logs, or test fixtures.
- Keep sensitive PendingAction/ActionReceipt confirmation, retrieval implementation, SSE, feedback, evaluation evidence, memory summarization, handoff, and later behavior out of this slice.

## Acceptance criteria

1. The CB-080 chat boundary still enforces direct-user identity, persisted session ownership, request idempotency, and ordered durable turn evidence before bounded agent execution; replay cannot create a second logical turn or re-execute a tool.
2. Exactly one ReAct agent owns clarification, refusal, and tool choice. `RuleRouter` emits only deterministic signals, while `ModelRouter` alone selects the business tier and escalation; no intent-classifier, decomposer, multi-agent, or proxy-side business routing is introduced.
3. One finite `attempt_budget` is created per accepted turn and charged consistently across application model calls, LiteLLM retry/fallback, HTTP, and tool attempts. Exhaustion returns a stable terminal denial/error and no layer can exceed or reset the shared budget.
4. LiteLLM receives role aliases only, performs at most one transient/network retry, and falls back only within the tier selected by `ModelRouter`. Non-transient provider errors, policy denials, schema errors, and exhausted budgets are not retried as network failures.
5. Provider circuit breakers are isolated by provider, require the configured minimum sample count before opening, have bounded open duration and half-open probes, and cannot cause cross-tier fallback or consume attempts without evidence.
6. Every executable tool is server-allowlisted by ToolSpec with fixed schema, risk, exact scope, timeout, idempotency, and model-visible output. Unknown tools, unknown fields, invalid arguments, scope widening, timeout widening, and unsafe output reject before commerce mutation.
7. The first commerce tool call obtains a just-in-time CB-020 OBO for the validated user/session and exact ToolSpec scope. Direct-user tokens, wrong issuer/audience/type/actor/scope/session, body identity substitution, cross-user resources, and unavailable/invalid exchange fail closed.
8. Expected adapter rejection produces a structured `deny_with_feedback` result that the same Agent may use only within the remaining budget. Unexpected programmer/configuration errors remain visible and are not converted into a permissive model response.
9. Real integration through the deterministic fake-provider/LiteLLM-compatible path proves model success, tool success, malformed model arguments, unknown tool, OBO/exchange denial, tool timeout, provider transient retry, same-tier fallback, circuit open/half-open recovery, budget exhaustion, and replay without duplicate tool execution.
10. Durable server-side evidence records the selected tier alias, charged attempts, routing signals, provider/circuit outcome, bounded tool lifecycle, denial, and terminal result in order, while public output and logs contain no raw token, secret, prompt, provider credential, private tool payload, or database error.
11. No real provider credential, retrieval/index behavior, sensitive PendingAction/ActionReceipt execution, SSE, feedback, evaluation evidence route, summary/memory packing, handoff, web, deployment, or later-slice behavior is introduced.

## Rejection paths

- Reject multi-agent/decomposer/intent-classifier behavior, model-selected scopes or tiers, proxy business routing, and deterministic signals treated as final policy.
- Reject independent or resettable retry counters, stacked unbounded retries, same attempt charged twice or not charged, retry after non-transient policy/schema errors, cross-tier proxy fallback, and success after budget exhaustion.
- Reject a circuit that opens before its minimum sample count, shares failure state across providers, probes without a bound, or silently bypasses an open provider without charging and recording the attempt.
- Reject client/model-defined tools, schemas, scopes, timeouts, risk tiers, identity, sessions, idempotency, or model-visible output; reject raw commerce responses passed directly to the model.
- Reject OBO acquisition before tool need, OBO cache keys broader than user/session/exact scope, token reuse beyond expiry, direct-user or service credential forwarded as commerce tool authority, and body identity substitution.
- Reject swallowed tool/provider/configuration failures, permissive fallback after authentication or ownership denial, and `deny_with_feedback` used to hide unexpected exceptions.
- Reject public or logged disclosure of prompts, credentials, token claims, raw provider/tool payloads, private evidence, circuit state internals, or database errors.
- Reject capability claims based only on unit mocks without deterministic proxy/fake-provider, real CB-020 exchange/session, commerce tool, durable event ordering, timeout, retry/fallback, circuit, exhaustion, replay, and rejection evidence.

## Out of scope

- retrieval, reranking, knowledge indexing, FAQ answer grounding, citation sufficiency, and Support Redis memory/summary behavior;
- sensitive PendingAction preparation/confirmation, ActionReceipt truth, refund or other later commerce behavior, and agent-side receipt commit semantics;
- SSE, feedback, evaluation evidence APIs, evaluation sandbox lifecycle, human handoff/tickets, web UI, deployment, real provider performance, and measured quality.

- real CB-020 session and JIT OBO evidence proving exact scope, token cache bounds, actor/audience/type/session/user validation, cross-user/body-identity denial, and no commerce authority before tool use;

- real `cs_db` evidence for ordered routing/model/budget/circuit/tool/denial/terminal events, transaction rollback, request replay, and public/log redaction;
- committed configuration and secret-scan evidence proving CI uses no real provider key and application code contains no concrete provider model identifier;
