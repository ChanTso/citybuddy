## Goal

Add two independently addressed, health-gated Redis runtimes with policies that match their distinct commerce and support workloads.

## In scope

- Extend the local Compose topology with two pinned Redis 7 images or digests.
- Configure Commerce Redis with `noeviction` and AOF persistence.
- Configure Support Redis for TTL-oriented data with an LFU eviction policy.
- Expose distinct required connection variables without adding fallback logic that can silently switch instances.
- Extend local startup, shutdown, health, and integration checks with runtime configuration and persistence evidence for both instances.

## Acceptance criteria

1. `make up` returns only after both Redis instances answer authenticated commands and report the required effective policies.
2. The commerce and support URLs resolve to different containers and data paths.
3. Runtime inspection proves Commerce Redis uses `noeviction` and AOF, while Support Redis uses its selected LFU policy and TTL-capable configuration.
4. A persistence probe survives the documented non-destructive restart path for Commerce Redis.
5. A controlled failure of either health check makes startup non-zero and visible.

## Rejection paths

- Reject a single Redis instance, identical policies for both workloads, or URLs that resolve to the same runtime.
- Reject application configuration that can silently fall back from one Redis role to the other.
- Reject process-only or port-only readiness where an authenticated Redis command is available.
- Reject treating either Redis instance as transaction, inventory, quota, action, or support-evidence truth.

## Out of scope

- MySQL identity or migration changes beyond consuming the verified CB-010 topology;
- Redis-backed business cache, lock, admission, session, or evidence behavior;
- Elasticsearch, IK, RocketMQ, APIs, and aggregate runtime closure.

## Required evidence

- Pinned Redis runtime configuration and image provenance;
- effective configuration output for both distinct instances;
- URL/container separation and Commerce Redis restart-persistence evidence;
- controlled health failure with non-zero startup;
- real Branch, PR, Commits, Tests, and Notes only at completion.
