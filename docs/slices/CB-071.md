## Goal

Add an owner-scoped, idempotent refund lifecycle and MySQL-authoritative payment/refund reconciliation that preserve the legal order, payment, refund, ledger, and inventory-restoration invariants as one coherent financial boundary.

## In scope

- Expose owner-scoped refund request and status contracts through the existing direct-user identity boundary. Derive ownership from the validated token, require idempotency, and re-read current MySQL order and payment truth before accepting a refund.
- Persist commerce-owned refund truth with a stable refund identifier, canonical request intent, request idempotency key, order/payment ownership, eligible amount, refunded amount, legal `REQUESTED → PROCESSING → SUCCEEDED / FAILED` transitions, and deterministic replay/conflict behavior.
- Enforce that cumulative successful refund amount never exceeds the eligible successfully paid amount. Unknown, unpaid, cancelled, already fully refunded, cross-user, amount-mismatched, or otherwise ineligible requests reject without a new refund or ledger movement.
- Apply each legal refund transition and its stable-key refund ledger movement in the required MySQL transaction boundary. Duplicate or reordered processing returns the existing state without another movement.
- Reconcile payment attempts, callback truth, order state, refunds, inventory ledger, and prior unpaid-timeout restoration from current MySQL truth. Reconciliation detects and reports inconsistent state and only performs explicitly legal, idempotent convergence; it never infers business truth from Redis or message payloads.
- Prove the cross-lifecycle invariant that unpaid cancellation, payment, and refund cannot restore inventory or activity quota twice. Refund money movement does not silently create an inventory restoration unless the current durable lifecycle explicitly requires one and a unique stable ledger key proves it exactly once.
- Keep refund request/processing independently callable by the commerce boundary needed before CB-120. Sensitive agent confirmation, PendingAction, ActionReceipt, and agent execution remain owned by later slices.
- Publish any refund domain event through the normal transactional Outbox: the refund mutation and Outbox row commit together, while delivery and replay cannot change the durable result.
- Extend only exact `commerce_app` DML grants required for refund and reconciliation table families; prove unrelated runtime identities remain denied.

## Acceptance criteria

1. Refund submission/status enforce direct-user issuer, audience, type, token-derived ownership, canonical intent, stable schema, and an owner/order-scoped idempotency key; body identity, cross-user access, and invalid token modes reject.
2. One eligible request creates one stable refund. Same-intent replay returns it, while conflicting key/intent reuse, an unknown or ineligible order/payment, invalid amount, and cumulative over-refund reject without another refund.
3. Refund state transitions are legal, conditional, and MySQL-authoritative. Duplicate, reordered, and late processing returns the existing state and cannot regress a terminal refund.
4. Each successful refund transition and exactly one stable-key refund ledger movement commit or roll back together. A failed refund records only the legal failure truth and never a successful movement.
5. Database failure rolls back the required refund/ledger/Outbox writes together; replay after an ambiguous result remains harmless through uniqueness, conditional state/version checks, and stable business-event keys.
6. Reconciliation compares payment attempt/callback, order, refund, inventory-ledger, and timeout-restoration truth in MySQL; it reports unresolved contradictions and applies only deterministic idempotent convergence allowed by the lifecycle.
7. Early, duplicate, reordered, and late timeout/payment/refund events cannot revive cancelled orders, over-refund payments, or restore inventory/activity quota twice. Stable ledger identities prove each permitted movement at most once.
8. Redis, RocketMQ delivery, event payloads, and in-memory state never override current MySQL payment/refund/order/ledger truth. Any cache or projection update follows durable commit or MySQL reconciliation.
9. Permission tests prove only required `commerce_app` DML, unrelated-runtime denial, runtime-DDL denial, and absence of bootstrap/migration or committed callback credentials in application configuration.
10. No real payment provider, chargeback/settlement operations, agent PendingAction/ActionReceipt, support-agent execution, evaluation sandbox, web UI, generalized recovery scanner, or later behavior is introduced.

## Rejection paths

- Reject wrong issuer/audience/type, body-supplied owner, cross-user order/refund access, missing/conflicting idempotency, unknown payment/order, invalid amount, cumulative over-refund, and unpaid/cancelled/ineligible lifecycle states.
- Reject illegal or regressive refund transitions, duplicate processing that writes again, terminal-state overwrite, and refund or ledger rows without stable replay identities.
- Reject refund and successful refund-ledger movement split across transactions, acknowledgement before commit, swallowed database errors, or retry that can create a second movement.
- Reject reconciliation that trusts Redis, message payload state, callback payload alone, or an in-memory result instead of current MySQL truth; reject silent repair of an ambiguous or contradictory lifecycle.
- Reject cancellation/payment/refund ordering that revives a cancelled order, refunds more than paid, restores inventory/activity quota twice, or mutates a paid/final order through a stale timeout.
- Reject refund-domain event publication outside the business transaction or an Outbox replay that mutates refund truth again.
- Reject direct agent execution, PendingAction/ActionReceipt behavior, sandbox behavior, external-provider integration, deployment, or later work implemented early.
- Reject capability claims based only on mocks without real MySQL lifecycle, duplicate/reordered event, rollback, reconciliation, stable-ledger, permission-denial, and controlled-failure evidence.

## Out of scope

- real payment-provider refund APIs, money settlement, chargeback, dispute, and provider webhook operations;
- agent confirmation, PendingAction, ActionReceipt, support conversation behavior, and sensitive tool execution;
- evaluation sandbox behavior, web UI, deployment, broad recovery scanning, and measured performance.

## Required evidence

- executable API/contract evidence for owner-scoped refund submission/status, same-intent replay, conflicting intent, wrong token modes, cross-user denial, eligibility, amount bounds, and stable schemas;
- real MySQL evidence for refund uniqueness, legal and illegal state transitions, cumulative amount bounds, one atomic successful refund/refund-ledger/Outbox boundary, rollback, terminal replay, and harmless retry after ambiguity;
- real reconciliation evidence spanning payment attempt/callback, order, refund, timeout restoration, inventory/activity-quota ledger, detected contradictions, and deterministic allowed convergence;
- duplicate, reordered, early, and late timeout/payment/refund evidence proving no cancelled-order revival, over-refund, duplicate money movement, or double inventory/activity-quota restoration;
- evidence that Redis and RocketMQ remain non-authoritative and that cache/projection changes follow committed MySQL truth;
- migration and permission output including unrelated-runtime and runtime-DDL denial;
- evidence that no external provider, agent action, sandbox, web, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
