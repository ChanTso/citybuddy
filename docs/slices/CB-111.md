## Goal

Consume the CB-110 normal knowledge-synchronization event in the Python knowledge-indexer and converge published FAQ versions and tombstones into the current versioned Elasticsearch projection without allowing duplicate, delayed, conflicting, or restarted delivery to regress public knowledge, while leaving the CB-112 cache and CB-113 through CB-115 rebuild/switch/rollback/cleanup outcomes unimplemented.

## In scope

- Add the production Python `knowledge-indexer` normal-message consumer for the fixed CB-110 knowledge event. Validate a closed, bounded envelope before any Elasticsearch mutation.
- Accept only the documented FAQ source type, positive monotonically comparable source version, explicit publication state, tombstone flag, stable event/source identities, bounded public content/reference, and occurred time. Reject unknown fields, private/draft data, malformed identifiers, invalid states/versions, and unbounded payloads.
- Compare every event with durable owner-local source-version/tombstone evidence. A newer published version idempotently upserts the deterministic FAQ document; an equal same-intent replay is a no-op; older or conflicting equal-version delivery cannot overwrite the projection.
- Apply a newer tombstone by removing the public FAQ document while retaining enough durable version/tombstone evidence to prevent an older publication from resurrecting it. Repeated and out-of-order tombstones are idempotent.
- Keep the existing `knowledge_docs_read` alias and current approved physical index as the search boundary. This slice performs incremental mutation only and does not create, validate, switch, or roll back a rebuild index.
- Keep source publication truth in `commerce_db`, the searchable projection in Elasticsearch, and historical turn evidence in `cs_db`. The indexer consumes the bounded public event and never reads or writes either database directly.
- Preserve CB-090 hybrid retrieval and CB-091 rerank/sufficiency/evidence behavior. Incremental projection changes may affect later searches by source version, but this slice does not add a cache, change ranking policy, rewrite stored evidence, or make Elasticsearch authoritative over commerce publication truth.
- Update the executable CB-104 asynchronous inventory for the new production-only knowledge consumer and prove no evaluation request can produce its FAQ publication event. Because the path remains evaluation-unreachable, this slice does not claim a sandbox liveness guard; any future change making it evaluation-reachable triggers the frozen introduction-point obligation.
- Use only the Elasticsearch access already assigned to `knowledge-indexer`. Add no MySQL or Support Redis grant, cross-database access, new external dependency, public FAQ authoring API, evaluator behavior, or web/deployment work.
- Use real RocketMQ, Elasticsearch, restart, redelivery, ordering, tombstone, outage, bounds, redaction, and no-direct-database-access evidence.

## Acceptance criteria

1. A production Python consumer receives the exact CB-110 normal-message envelope through the real Broker, validates its closed bounded schema, and applies only supported FAQ publication/tombstone events.
2. A newer published FAQ version idempotently upserts one deterministic document in the current aliased physical index. Equal same-intent replay makes no duplicate or content change; an older version is acknowledged as stale without regressing the document.
3. Reuse of an event identity or equal source version with different source, state, tombstone, content/reference, or occurred-time intent fails closed and cannot overwrite accepted projection/version evidence.
4. A newer tombstone removes the FAQ from public search and durably fences its source version. Duplicate/older tombstones are harmless, and later delivery of any older publication cannot resurrect the FAQ.
5. Broker redelivery, consumer restart after accepted mutation but before acknowledgement, and repeated delivery converge to one projection/version outcome without duplicate documents, version regression, or unbounded retry.
6. Elasticsearch unavailability or indeterminate mutation is never acknowledged as applied. Recovery and redelivery converge from durable projection/version evidence; malformed or permanently rejected messages follow a bounded owner-local rejection disposition without mutating public knowledge.
7. The existing read alias remains unchanged and every incremental write stays within its current approved physical index. No CB-113 rebuild/validation/switch, CB-114 rollback, or CB-115 cleanup behavior is implemented early.
8. Existing hybrid retrieval, rerank, calibrated sufficiency, citations, shared attempt budget, and persisted retrieval-evidence semantics remain unchanged. Repeated historical evidence reads never re-query Elasticsearch or adopt a newer FAQ version retroactively.
9. The updated executable asynchronous inventory identifies the knowledge consumer as production-only and proves the real FAQ publication entry is evaluation-unreachable. No sandbox field/property, evaluation producer, or runtime liveness-guard claim is added.
10. Exact-access evidence proves `knowledge-indexer` can mutate only its required Elasticsearch projection and cannot directly access `commerce_db`, `cs_db`, or Support Redis; application identities retain their existing database boundaries and runtime DDL/admin access remains denied.
11. Real integration covers valid publish, revision, duplicate, conflicting identity/version, older delivery, tombstone, post-tombstone stale publication, controlled pre-ack failure, redelivery, consumer restart, Elasticsearch outage/recovery, bounds, redaction, unchanged retrieval evidence, and production-only reachability.
12. No versioned two-level FAQ cache, full rebuild/alias handoff, public FAQ authoring API, withdrawal source transition beyond consuming the frozen tombstone contract, retrieval-policy change, evaluator, action/receipt, memory, handoff, web, deployment, real-provider access, or measured performance claim is introduced.

## Rejection paths

- Reject malformed or oversized envelopes, unknown fields/source types/states, nonpositive versions, draft/private content, arbitrary metadata, caller-selected index keys, credentials, SQL, internal endpoints, and raw errors.
- Reject older publication or tombstone delivery that regresses a newer watermark, equal-version conflicting content, event-identity rebinding, duplicate documents, and resurrection after a newer tombstone.
- Reject acknowledging an indeterminate Elasticsearch mutation as applied, unbounded poison-message retry, or Elasticsearch state used to repair or replace authoritative commerce publication truth.
- Reject direct indexer reads from `commerce_db`/`cs_db`, any Support Redis access, broad grants, runtime DDL/admin access, request-thread dual writes, and independent MySQL/RocketMQ/Elasticsearch writes that can diverge.
- Reject evaluation reachability inferred from mocks, direct repository calls, database fixtures, or hand-built events. If a real evaluation producer is introduced, reject completion without the same-slice sandbox guard and full real-Broker evidence required by the frozen contract.
- Reject CB-112 query/answer cache, CB-113 through CB-115 rebuild/switch/rollback/cleanup behavior, public FAQ authoring, new withdrawal source truth, evaluator/actions/memory/handoff/web/deployment work, or measured performance claims implemented early.
