## Goal

Provide a reproducible, health-gated MySQL foundation with independent migration histories and least-privilege identities. Establish explicit synthetic local credential initialization without containerizing APIs or creating business schemas.

## In scope

- Add a Docker Compose topology for one pinned MySQL 8 image with `commerce_db` and `cs_db`.
- Use a bootstrap/admin identity only for database, account, and grant creation. Put MySQL's required delegated privileges and `GRANT OPTION` in a dedicated non-default grant role, keep `activate_all_roles_on_login=OFF`, and add a separately invokable one-shot grant job. The job verifies `CURRENT_ROLE()=NONE`, explicitly activates the role only while applying the repository's fixed version-controlled allowlist of exact `GRANT`/`REVOKE` statements, returns the session to `NONE`, rejects caller-supplied SQL and business-data DML, and does not expose bootstrap credentials or the role to migrations or applications. Add separate auth, commerce, and agent migration identities plus runtime identities `auth_app`, `commerce_app`, and `agent_app`.
- Establish separately invokable, rerunnable migration jobs for the auth-owned and commerce-owned streams in `commerce_db` and the agent-owned stream in `cs_db`. Each stream records real independent history and fails if it cannot validate its target database.
- Treat migration completion as a dependency gate that later API services must use when they are introduced. Do not package or add API/worker containers in this slice, and do not make migrations an application-startup side effect.
- Add an explicit `make init-local` path that creates an ignored local environment file with fresh synthetic credentials on first use. It must not overwrite or rotate an existing file implicitly. `.env.example` lists required variable names and safe descriptions but contains no credential values; runtime commands do not use hidden fallback credentials.
- Add meaningful MySQL and migration-completion health gates plus real `make up`, `make down`, and MySQL-focused `make test-integration` behavior.
- Test all declared identities and present denial boundaries. Do not create placeholder business tables merely to test grants.

## Acceptance criteria

1. From a clean checkout, `make init-local` creates the ignored environment file without committed values; a repeat run preserves it, and an explicit reset or removal is required before rotation.
2. `make up` fails clearly when required local configuration is absent and otherwise returns only after MySQL is ready for authenticated queries and all three migration streams complete successfully.
3. `make down` removes this slice's containers predictably without deleting durable data.
4. Bootstrap, three migration identities, and three runtime identities are distinguishable. Migration jobs are independently invokable and rerunnable and maintain separate histories.
5. `activate_all_roles_on_login=OFF`, and a fresh bootstrap session reports `CURRENT_ROLE()=NONE`. Only the separately invoked one-shot grant job activates the bootstrap grant role, executes the repository's fixed version-controlled allowlist of exact `GRANT`/`REVOKE` statements, returns the session to `NONE`, and rejects caller-supplied SQL, business-data DML, and any non-allowlisted statement, object, or privilege.
6. `agent_app` cannot access `commerce_db`; runtime identities cannot execute DDL or use global/admin grants; no application or migration configuration uses bootstrap/admin credentials, the bootstrap grant role, or another stream's migration credentials.
7. Auth-versus-commerce family denials are deferred until real owning tables exist in `CB-020` and `CB-030`; this slice does not invent placeholder tables for those tests.
8. A controlled MySQL health, grant-job, or migration failure makes startup non-zero and visible.

## Rejection paths

- Reject fixed-sleep startup, process-only health, or a command that masks an unhealthy database or failed migration.
- Reject committed/generated credentials in tracked files, implicit default passwords, silent credential rotation, or an environment example containing usable values.
- Reject application or migration use of bootstrap credentials or the bootstrap grant role, `activate_all_roles_on_login=ON`, a fresh bootstrap session with an active role, caller-supplied or non-allowlisted grant-job SQL, a non-allowlisted target or privilege, business-data DML in the grant job, runtime DDL/global/admin grants, `agent_app` access to `commerce_db`, or cross-database joins.
- Reject an API/worker container added merely to host migrations, schema mutation during normal application startup, or a migration target that succeeds without validating its database.
- Reject placeholder business tables created only to make a permission test possible.

## Out of scope

- Redis, Elasticsearch, IK, RocketMQ, and aggregate runtime closure;
- API or worker containerization and runtime packaging;
- complete auth, commerce, or agent business schemas and seed data;
- login, JWT/OBO, transaction, agent, retrieval, evaluation, or web behavior.

## Required evidence

- Pinned MySQL Compose configuration and image provenance;
- clean `make init-local`, repeat-run preservation, missing-config rejection, `make up`, health, migration, and `make down` output;
- repeat-run logs for each independent migration history;
- account/role/grant output proving `activate_all_roles_on_login=OFF`, a fresh session starts at `CURRENT_ROLE()=NONE`, the one-shot job activates the grant role only for the fixed version-controlled allowlist and returns to `NONE`; rejection evidence for caller-supplied or non-allowlisted SQL, business DML, default role auto-activation, and bootstrap credentials or role appearing in migration/runtime configuration; plus denials proving runtime identities lack DDL/global/admin privilege and `agent_app` cannot access `commerce_db`;
- controlled health, grant-job, or migration failure with non-zero startup;
- real Branch, PR, Commits, Tests, and Notes only at completion.
