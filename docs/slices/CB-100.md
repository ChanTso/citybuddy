## Goal

Add the evaluation-profile-only auth boundary for service-authenticated test-principal provisioning and revocation, opaque-handle test-token issuance, and unchanged sandbox binding on derived OBO tokens, without implementing the commerce sandbox lifecycle or an evaluator.

## In scope

- Add an auth-owned TTL-bound evaluation test-principal provisioning record in `commerce_db`. Its server-generated opaque handle is uniquely bound to one sandbox id and case correlation, and provisioning/revocation idempotency is persisted rather than inferred from process memory.
- Add evaluation-profile-only `POST /internal/eval/test-principals/provision` for a dedicated authenticated `commerce-service` client. Validate bounded sandbox id, case correlation, TTL, minimum test-subject attributes, and idempotency before writing; same-intent retry returns the same valid handle and conflicting reuse rejects.
- Add evaluation-profile-only `POST /internal/eval/test-principals/{handle}/revoke` for the same dedicated service identity. Revocation is idempotent, preserves the original sandbox/case binding, and prevents subsequent token issuance from the handle.
- Add evaluation-profile-only `POST /auth/eval/test-token` behind an independent evaluation API credential. It accepts only the opaque handle plus matching `X-Eval-Sandbox-Id`, validates the auth-owned provisioning record and expiry/revocation state, and returns an explicitly typed, bounded-lifetime RS256 direct-user test JWT carrying the bound sandbox claim.
- Extend the existing JIT token exchange only so a valid evaluation direct-user JWT produces an otherwise unchanged exact-scope OBO with the identical sandbox claim. A production token cannot acquire a sandbox claim, an evaluation token cannot drop or replace it, and neither request fields nor the authenticated service can select a different sandbox.
- Keep signing keys, issuer/audience/type, principal state, service authentication, session-binding assertion, exact ToolSpec scope, actor, time bounds, JWKS overlap, and private-key handling inside the existing CB-020 identity boundary.
- Extend only the exact auth migration/runtime privileges required for the auth-owned provisioning record. `auth-service` never reads the commerce sandbox registry or other commerce business tables; `commerce-service` and `agent-service` never read the provisioning table directly.
- Keep every new route unavailable outside the evaluation profile. Production cannot issue evaluation tokens and continues to reject evaluation sandbox context.
- Use deterministic local credentials and real JWT/MySQL evidence. Do not introduce ServiceEval, commerce reset/activation/completion, sandbox business-table filtering, asynchronous liveness guards, evaluation state/evidence APIs, or irreversible external effects.

## Acceptance criteria

1. The auth migration creates one auth-owned provisioning record with a server-generated opaque handle, unique sandbox/case and idempotency invariants, bounded expiry, explicit provisioned/revoked state, and no stored or returned reusable user credential.
2. Provisioning is loaded only by the evaluation profile and accepts only the dedicated authenticated `commerce-service` identity. Same-intent retries return the same valid handle; conflicting correlation/idempotency reuse, invalid TTL/attributes, wrong service identity, and production-profile access reject without a record.
3. Revocation is evaluation-profile-only, dedicated-service-authenticated, and idempotent. It rejects a mismatched handle/sandbox/case binding or conflicting key and leaves the original record unchanged; an already revoked binding remains revoked.
4. Test-token issuance requires an independent evaluation API credential, an opaque handle, and an exactly matching sandbox header. Unknown, expired, revoked, cross-sandbox, malformed, or conflicting handles reject without revealing whether another sandbox owns the record.
5. A successful test token is RS256 signed, explicitly typed as a direct-user evaluation token, uses the fixed issuer and configured user-facing audience, contains the server-owned subject and sandbox claim, has bounded time claims no later than the provisioning expiry, and contains no password, service credential, provisioning idempotency key, or private metadata.
6. Existing token exchange preserves the exact sandbox claim from a valid evaluation direct-user token into the short exact-scope OBO while retaining the CB-020 subject, session, audience, scope, actor, type, and time-bound rules. Claim insertion, removal, substitution, header mismatch, production-to-evaluation upgrade, and evaluation-to-production downgrade fail closed.
7. `auth-service` decides provisioning, revocation, and issuance only from its auth-owned record and never reads the commerce sandbox registry. Fixture creation, activation, completion, compensation, and janitor behavior remain unimplemented until CB-101; asynchronous sandbox liveness remains unimplemented until CB-104.
8. Exact-grant evidence proves only required `auth_app` DML on the provisioning record, auth migration ownership, `commerce_app`/`agent_app` denial, runtime-DDL denial, and no direct access from auth runtime to commerce business or `cs_db` tables.
9. Production-profile evidence proves all three evaluation auth routes are not loaded, production direct-user tokens carry no sandbox claim, production cannot issue test tokens, and evaluation headers do not become an authentication fallback.
10. Real integration proves provision/retry/conflict, revoke/retry, expiry, token issuance and claim validation, unchanged OBO sandbox propagation, key/JWKS validation, restart-stable lifecycle truth, permission denials, and private-field redaction without external evaluator or provider dependencies.
11. No commerce sandbox registry/reset/fixtures/activation/completion, business-table sandbox filtering, asynchronous liveness, evaluation state/audit/version/evidence API, mock-payment sandbox callback, ServiceEval implementation, web, deployment, or later behavior is introduced.

## Rejection paths

- Reject production-profile access; missing, invalid, wrong-service, or substituted internal/evaluation credentials; and any anonymous or broader-token fallback.
- Reject client-selected handles or subjects, credentials returned as handles, unbounded/expired TTL, malformed sandbox/case/attribute values, same key with different intent, and duplicate records for one provisioning identity.
- Reject unknown, expired, revoked, cross-sandbox, cross-case, or mismatched handles for token issuance and revocation without disclosing another binding's private state.
- Reject a test token with wrong issuer/audience/type, missing or caller-selected sandbox, lifetime beyond record expiry, private provisioning fields, service credentials, or signing material.
- Reject OBO sandbox insertion, stripping, replacement, header/claim mismatch, scope widening, actor/session substitution, production/evaluation claim-mode downgrade, and caching across sandbox boundaries.
- Reject `auth_app` access to commerce sandbox/business truth, `commerce_app`/`agent_app` access to provisioning records, runtime DDL, broad grants, committed evaluation credentials, or secret/private-data logging.
- Reject commerce reset/activation/completion, sandbox liveness/filtering, evaluator/evidence APIs, irreversible-effect behavior, or later work implemented early.
- Reject lifecycle and token claims based only on unit mocks without real MySQL persistence, restart, RS256/JWKS, service/evaluator authentication, OBO propagation, permission, production-profile, and redaction evidence.

## Out of scope

- commerce-owned sandbox registry, reset-created fixtures, `PROVISIONING → ACTIVE → DEAD`, activation/compensation/completion, TTL janitor, and business repository filtering;
- evaluation state, audit, version, support-evidence APIs, async consumer liveness, sandboxed mock-payment callbacks, and ServiceEval implementation;
- retrieval, actions/receipts, memory, handoff, web UI, deployment, real-provider access, and measured performance.

## Required evidence

- executable OpenAPI/profile/contract tests for the three auth routes, dedicated service and evaluator credential chains, bounded schemas, stable errors, route absence in production, and private-field response/log allowlists;
- real auth migration and MySQL evidence for opaque-handle uniqueness, sandbox/case/idempotency binding, expiry/revoke lifecycle, same-intent retry, conflicting reuse, restart persistence, exact `auth_app` grants, unrelated-runtime denial, and runtime-DDL denial;
- real RS256/JWKS evidence for fixed issuer/audience/direct-user type, server-owned subject, sandbox claim, expiry capped by provisioning TTL, key overlap, invalid credential/handle/header rejection, and no private signing material;
- real JIT exchange evidence proving identical sandbox propagation into exact-scope OBO and fail-closed insertion/removal/substitution, claim-mode, scope, actor, subject, and session rejection;
- evidence that auth never reads commerce sandbox/business truth and no reset/fixtures/activation/completion, repository filtering, async liveness, evaluation evidence/state API, evaluator, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
