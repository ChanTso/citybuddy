## Goal

Create the first versioned public-knowledge Elasticsearch index and stable read alias, then expose `knowledge.search` with separate BM25 and dense recall plus deterministic application-side reciprocal-rank fusion, without adding reranking, sufficiency calibration, incremental publication, or rebuild automation early.

## In scope

- Create one physical `knowledge_docs_vN` index behind a stable read alias. The mapping supports the frozen shared FAQ/product document model, `doc_type`, source identity/version, publication/deletion metadata, analyzed text, and the pinned dense-vector shape required by real kNN queries.
- Load a bounded deterministic public-knowledge corpus sufficient to prove the initial searchable index. Private orders, refunds, coupons, credentials, support evidence, user identity, sandbox data, and other transactional or personal truth never enter Elasticsearch.
- Keep Elasticsearch derived and non-authoritative. Live price, stock, availability, payment, refund, and order state remain commerce-owned and cannot be answered from indexed fields as current transaction truth.
- Add the server-owned `knowledge.search` tool boundary in `agent-service`. The request keeps the original query and an optional rewrite as distinct recall inputs; neither may carry owner, index, filter, candidate-count, score, or source authority supplied by the model or caller beyond the fixed ToolSpec schema.
- Execute separate bounded BM25 and dense-vector recall for each allowed query input, normalize candidates into one fixed internal schema, and merge them with deterministic application-side RRF. Elasticsearch server-side RRF is not assumed or required.
- Deduplicate fused candidates by stable document/chunk identity, apply deterministic tie-breaking, bound every recall list and final result list, and return only allowlisted public source fields plus the index version actually queried.
- Fail closed when the alias is missing or ambiguous, the mapping/version is incompatible, embedding shape is invalid, Elasticsearch is unavailable, candidate payloads are malformed, or a result would expose non-public metadata.
- Use deterministic embedding/query fixtures or an equivalent local test path for repeatable CI evidence. Do not add a real provider credential, reranker, calibrated sufficiency decision, retrieval evidence persistence, FAQ publication consumer, incremental sync, tombstone processing, or automated rebuild/switch behavior.

## Acceptance criteria

1. A real Elasticsearch physical index named in the `knowledge_docs_vN` family has the required analyzed-text, dense-vector, source/version, publication/deletion, document-type, and public metadata mapping, and one stable read alias resolves to exactly that approved version.
2. The initial corpus contains only bounded public FAQ/product knowledge with stable source and chunk identities. Executable rejection evidence proves private commerce, user, support, credential, sandbox, and evaluation fields cannot enter or leave the index boundary.
3. `knowledge.search` is server-owned, bounded, and schema-validated. It preserves the original query and optional rewrite as separate inputs and rejects unknown fields, caller/model-selected index names, unbounded limits, unsupported filters, and malformed queries before Elasticsearch I/O.
4. Each allowed query input performs separate bounded BM25 and dense recall against the stable alias. A real mixed-language lexical query and a real kNN query return the expected public candidates from the pinned runtime.
5. Application-side RRF merges the independent ranked lists deterministically with a fixed constant, stable identity deduplication, stable tie-breaking, and bounded output. Repeated inputs return the same order, and server-side Elasticsearch RRF is not required.
6. Results expose only fixed public fields, stable source/chunk identity, source version, document type, bounded score/rank metadata, and the resolved index version. They do not expose raw vectors, query embeddings, internal Elasticsearch payloads, private metadata, credentials, or database details.
7. Missing/multiple aliases, incompatible mapping or vector dimensions, deleted/unpublished records, malformed candidate payloads, Elasticsearch timeout/unavailability, and partial recall failure terminate with bounded structured failure rather than unsupported or partial success.
8. Indexed price, stock, availability, order, payment, or refund data is never treated as live transaction truth; the search tool cannot perform commerce reads or writes and cannot acquire or forward OBO authority.
9. Real integration proves physical-index creation, mapping checks, stable alias resolution, BM25 recall, dense recall, original-plus-rewrite contribution, deterministic RRF/dedup/tie order, bounded output, redaction, and controlled failure cleanup.
10. No reranker role, sufficiency threshold/margin calibration, retrieval evidence persistence, incremental FAQ/product event consumer, version-order/tombstone mutation, full rebuild automation, alias rollover workflow, Support Redis retrieval cache, or later behavior is introduced.

## Rejection paths

- Reject private or transactional records, unpublished/deleted records, credentials, user/session/sandbox/evaluation fields, and any claim that Elasticsearch is live commerce truth.
- Reject model/client-selected alias or physical index, caller-controlled candidate sizes or RRF parameters, unknown request fields, empty/oversized/malformed queries, invalid rewrite shape, and unsupported metadata filters.
- Reject unbounded BM25, kNN, per-query, or final result counts; duplicate identities emitted twice; nondeterministic tie order; raw-score concatenation presented as RRF; and dependence on undeclared server-side RRF support.
- Reject raw vectors, embeddings, Elasticsearch request/response bodies, internal exceptions, private metadata, or credentials in tool output or logs.
- Reject missing or multi-target alias, wrong index-version family, mapping/vector mismatch, deleted/unpublished candidate leakage, malformed source records, and one recall leg failing while the tool reports an unqualified success.
- Reject OBO acquisition, commerce tool execution, reranking, sufficiency/answerability claims, retrieval-evidence writes, incremental publication/tombstones, rebuild automation, cache layers, or later work implemented early.
- Reject capability claims based only on unit fakes without the pinned Elasticsearch/IK runtime, real index/alias, real BM25 and kNN recall, deterministic fusion, failure, privacy, and cleanup evidence.

## Out of scope

- reranking, development-set sufficiency calibration, answer grounding policy, citation sufficiency, and durable retrieval evidence;
- FAQ draft/publish truth, incremental RocketMQ synchronization, source-version/tombstone mutation, two-level FAQ cache, full rebuild validation, alias rollover, rollback, and cleanup authorization;
- live commerce reads/writes, OBO acquisition, model-provider calls, agent-loop changes, SSE, feedback, evaluation, memory, handoff, actions, web, deployment, and measured retrieval quality or latency.

## Required evidence

- executable mapping and contract tests for `knowledge_docs_vN`, stable alias, required public fields, dense-vector dimensions, ToolSpec/request bounds, response allowlist, and forbidden private fields;
- real pinned Elasticsearch/IK integration proving physical-index creation, alias resolution, mixed-language BM25, dense kNN, original and rewrite recall, deleted/unpublished exclusion, and deterministic cleanup;
- deterministic application-side RRF evidence for fixed parameters, independent lists, stable deduplication/tie-breaking, bounded results, repeatability, and no server-side RRF dependency;
- controlled missing/multiple-alias, mapping/vector mismatch, malformed candidate, timeout/unavailability, partial-recall, unknown-field, oversized-query/limit, and raw-payload-redaction rejection evidence;
- evidence that Elasticsearch remains derived public knowledge, no private/live commerce truth or credentials are indexed, no OBO/commerce authority is used, and no CB-091/111/112 or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
