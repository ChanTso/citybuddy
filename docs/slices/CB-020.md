## Goal

Establish the production identity vertical slice from login and JWKS through server-owned support sessions and just-in-time, exactly scoped agent OBO exchange, with fail-closed commerce authorization and least-privilege persistence evidence.

## In scope

- Add auth-owned migrations in `commerce_db` for user principals, credential verifiers, independently revocable service identities, and signing-key metadata, using only the existing auth migration/runtime identities.
- Implement `auth-service` login, public JWKS, and authenticated token exchange boundaries with RS256 signing material supplied only at runtime.
- Distinguish direct-user and OBO tokens explicitly and enforce fixed issuer, intended audience, time bounds, principal state, purpose/type, and route-required authority.
- Implement `POST /api/sessions` in `agent-service` so it validates a direct-user JWT, generates an opaque support-session id server-side, and persists the immutable user binding in `cs_db`.
- Require every support-session use to recheck the persisted owner binding; reject client-selected identity, forged sessions, cross-user reuse, wrong token type, and production evaluation headers or claims.
- Issue a short-lived OBO only after `agent-service` independently authenticates, presents a validated direct-user token, asserts a verified support-session binding, and requests one server-allowlisted exact scope.
- Validate the OBO in `commerce-service` using fixed issuer, explicit OBO type, `aud=commerce-service`, exact scope, `act.azp=agent-service`, user subject, support session, and time bounds, without trusting request-body identity.
- Prove bounded unknown-`kid` JWKS refresh behavior and signing-key overlap sufficient for already-issued token validation.
- Publish executable API/security contracts and deterministic integration evidence without adding business endpoints or evaluation-profile provisioning.

## Acceptance criteria

1. An active test principal with valid credentials receives an RS256 direct-user JWT whose explicit type, issuer, user-facing audience, subject, authority, and time claims validate; invalid credentials and disabled principals reject without credential disclosure.
2. JWKS exposes only current and overlapping public keys with stable `kid` values; private signing material is absent from responses, persistence, logs, committed examples, and build artifacts.
3. Direct-token validation performs at most one JWKS refresh and one retry for an unknown `kid`, accepts a legitimately overlapping key, and rejects continued unknown, malformed, expired, premature, wrong-issuer, wrong-audience, or wrong-type tokens.
4. `POST /api/sessions` accepts only a valid direct-user token, creates an opaque server-generated id, and persists its immutable subject binding in `cs_db`; a client-supplied owner, OBO token, forged id, or cross-user reuse rejects.
5. Production session and token paths reject evaluation sandbox headers or claims. Evaluation principal provisioning, test-token issuance, and sandbox-bound extensions remain unimplemented until `CB-100`.
6. Token exchange authenticates `agent-service` independently, verifies the direct-user token and asserted persisted support-session binding, permits only a server-side exact-scope allowlist, and returns a short-lived OBO containing the required type, subject/user, support session, `aud=commerce-service`, exact scope, actor, `jti`, and time claims.
7. Exchange rejects an invalid/revoked service credential, wrong direct-token mode or audience, forged/cross-user session binding, missing or widened scope, arbitrary actor, and identity supplied only in a request body.
8. The commerce authorization boundary accepts the valid OBO chain and rejects a direct token or any wrong issuer, audience, type, scope, actor, subject/session, time bound, or body-identity substitution before business logic. No business endpoint is added solely to manufacture this evidence.
9. Executable permission checks prove `auth_app` can access only the auth-owned table family, `agent_app` can access only the support-session family in `cs_db`, and neither gains commerce business-table, cross-database, DDL, global, admin, bootstrap, or signing-secret access.
10. Contract and integration tests exercise the complete login → session → exchange → commerce-authorization chain plus the listed rejection paths with deterministic local credentials and no committed secret.

## Rejection paths

- Reject a shared signing key outside `auth-service`, symmetric signing, private key exposure through JWKS or persistence, unbounded key refresh, or key rotation that invalidates still-live tokens.
- Reject token-mode inference from a missing claim, permissive issuer/audience defaults, authentication fallback to anonymous, or direct-user acceptance on an OBO-only boundary.
- Reject token exchange based on an unauthenticated caller, a request-body user/session, an unverified support-session assertion, a model-selected scope, wildcard scope, or caller-requested actor.
- Reject client-generated support-session ownership, session lookup without subject revalidation, cross-user reuse, or production acceptance of evaluation sandbox context.
- Reject runtime use of bootstrap/migration credentials, auth access to commerce business tables, agent access to `commerce_db`, or secrets committed as convenient local defaults.
- Reject product, order, chat orchestration, tool execution, evaluation provisioning, or sandbox-liveness behavior added to demonstrate identity.

## Out of scope

- Product/profile APIs, orders, seckill, payment/refund, and other commerce business behavior;
- conversation turns, model access, tools, SSE, feedback, or support evidence beyond the support-session identity foundation;
- evaluation reset, principal provisioning/revocation, test-token issuance, sandbox registry/liveness, and sandbox-bound OBO extension owned by `CB-100`;
- production deployment, external identity providers, password recovery, MFA, federation, and measured security or performance claims.

## Required evidence

- auth and support-session migration ownership plus positive and negative database permission output;
- login and JWKS success, invalid/disabled-principal rejection, public-only key material, bounded unknown-`kid` refresh, and overlapping-key validation;
- server-generated support-session creation and forged, wrong-type, cross-user, and production-sandbox rejection;
- independently authenticated exact-scope exchange and invalid-service, forged-binding, scope-widening, and wrong-token-mode rejection;
- valid commerce OBO authorization and direct-token, issuer/audience/type/scope/actor/session/time/body-identity rejection;
- complete deterministic chain output through the real service boundaries, a passing real `make ci`, and no committed secret;
- real Branch, PR, Commits, Tests, and Notes only at completion.
