## Goal

Establish commerce-owned product and CRM truth with authenticated published-product reads, a non-authoritative Commerce Redis cache, and a transactional Outbox plus idempotent invalidation path that preserves MySQL authority under cache misses, duplicates, stale delivery, and controlled failures.

## In scope

- Add commerce-owned migrations in `commerce_db` for one profile per immutable user subject, stable products with monotonically increasing publication versions, and the first transaction Outbox records, using only the existing commerce migration and runtime identities.
- Define executable OpenAPI and contract tests for `GET /api/products` and `GET /api/products/{productId}`. Apply the CB-020 direct-user JWT policy unless the implementation explicitly selects and tests the frozen public-read alternative; never infer identity from request data.
- Return only published product data while keeping price, stock, availability, and publication state authoritative in MySQL.
- Implement Commerce Redis product caching with jittered TTLs, bounded null caching, Bloom-assisted penetration protection, and mutex-protected hot-key rebuild. A missing, stale, incomplete, or unavailable Redis structure must not create product truth or override MySQL.
- Make each product publication change and its uniquely identified, versioned Outbox event one MySQL transaction. A rollback leaves neither a changed product nor an orphan event.
- After a committed request-side product change, attempt best-effort cache deletion without converting cache success into transaction truth. The durable Outbox remains the recovery path when deletion fails.
- Publish normal commerce domain events from the Outbox and process duplicate or reordered invalidation events idempotently. Consumers delete or rebuild only from current MySQL state and cannot regress a newer cached version.
- Keep request threads from independently dual-writing MySQL, Redis, RocketMQ, or Elasticsearch. This slice may emit the versioned public product event boundary but does not implement the later production knowledge projection.
- Prove least-privilege separation: `commerce_app` can access the new commerce families, while `auth_app` and other unrelated runtime identities cannot read or mutate them.

## Acceptance criteria

1. Commerce migrations create the profile, product, and Outbox families with stable identifiers, required uniqueness, monotonically increasing product publication versions, and no access widening outside the commerce identities.
2. The product list and detail contracts return published MySQL-backed product data with live commerce fields; missing or unpublished products and invalid identity modes follow the documented rejection semantics.
3. Repeated nonexistent reads are bounded by Bloom/null-cache protection without allowing Redis to manufacture an existing or missing product result when its filter/cache state is absent, stale, or unavailable.
4. Concurrent hot-key misses perform one mutex-protected rebuild per cache key/window, populate a jittered TTL from current MySQL state, and do not expose a partial cache value.
5. A product publication mutation and its unique versioned Outbox record commit or roll back together. Request-side cache deletion is best effort after commit, and its controlled failure leaves a durable recovery event.
6. Duplicate event delivery is harmless, and an older product version cannot delete or rebuild over a newer cached projection. Consumer retry obtains authoritative state from MySQL.
7. Controlled cache outage, cache-delete failure, publisher failure, consumer failure, duplicate delivery, and stale delivery remain visible and preserve MySQL truth without a false success or data regression.
8. `auth_app` is denied reads and writes to the product, CRM, and Outbox families; `commerce_app` has only the runtime DML required by this slice and cannot perform DDL or access auth-private tables.
9. No cart, order, seckill, payment, agent tool, Elasticsearch product projection, FAQ publication, or web product experience is introduced.

## Rejection paths

- Reject a cache hit, Bloom result, lock acquisition, or RocketMQ acknowledgement treated as authoritative product or CRM truth.
- Reject a Bloom negative used as a durable not-found answer when the filter is uninitialized, incomplete, stale, or cannot prove it represents the current published set.
- Reject unbounded lock waits, fixed-sleep correctness, stampede-prone rebuilds, non-jittered universal TTLs, or null entries that can outlive the bounded penetration-protection policy.
- Reject a product mutation committed without its Outbox row, an Outbox row committed for a rolled-back mutation, or a request reported failed solely because post-commit best-effort cache deletion failed.
- Reject duplicate or older events that regress cache state, rebuild from event payload as truth instead of current MySQL state, or silently acknowledge a failed invalidation.
- Reject request-thread Elasticsearch writes, direct knowledge-indexer database access, or independent MySQL/cache/search writes whose partial success can diverge.
- Reject public exposure of unpublished products, identity fallback after JWT validation failure, request-body identity substitution, or widened database grants.
- Reject capability claims based only on unit mocks without real MySQL, Redis, RocketMQ, permission-denial, duplicate, stale-event, and controlled-failure evidence.

## Out of scope

- Cart, standard ordering, inventory mutation, seckill, payment, refund, and inventory-ledger behavior;
- agent ToolSpecs, OBO commerce tools, support-session behavior, and CRM use by the agent;
- production FAQ workflow, knowledge-indexer implementation, Elasticsearch product indexing, RAG, and alias rebuilds;
- web UI, production deployment, performance, resilience, and operational-readiness claims.

- evidence that request paths do not dual-write Elasticsearch and that no later business slice was implemented early;
