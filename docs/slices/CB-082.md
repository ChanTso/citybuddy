## Goal

Expose the CB-080/CB-081 durable support turn through a strictly filtered SSE projection, add owner-scoped append-only feedback, and prove the support path end to end with deterministic model fakes without leaking internal agent, tool, retrieval, token, or database evidence.

## In scope

- Add `POST /api/chat/stream` with the same direct-user JWT, persisted support-session ownership, request correlation/idempotency, and production-evaluation-context rejection rules as `POST /api/chat`. The streaming route reuses the CB-080 durable turn and CB-081 bounded-agent boundaries rather than creating a second conversation truth.
- Keep internal ordered agent events authoritative in `cs_db` and project them through one server-owned SSE egress filter. Public event names are limited to `token`, `action_receipt`, `done`, and `error`; each event has a fixed allowlisted schema and cannot carry raw internal event payloads.
- Buffer ordinary text through the frozen small output-safety boundary before emitting it. Action claims such as an order or refund success are not released as ordinary tokens without matching successful receipt truth; this slice does not invent or synthesize ActionReceipt truth before CB-120/CB-121.
- Preserve ordered, single-terminal stream semantics. A stream cannot emit after `done` or terminal `error`, expose an internal retry as a second logical answer, or return success before the accepted turn has the durable outcome/evidence required by CB-080 and CB-081.
- Handle disconnect, cancellation, timeout, and bounded backpressure without changing durable turn truth, leaking internal exceptions, or starting unbounded background work. Replay uses the persisted turn/correlation boundary and cannot re-execute a completed tool side effect.
- Add owner-scoped `POST /api/feedback`. Re-read the persisted user/session/trace association, generate the feedback identity on the server, persist an append-only user signal in `cs_db`, and make same-intent retry deterministic while conflicting reuse rejects.
- Exercise the complete support path with deterministic fake-provider scenarios: direct-user/session validation, bounded model routing, safe text streaming, tool success and denial, terminal failure, replay, feedback, and server-side ordered evidence. CI contains no real provider credential or network dependency.
- Extend only exact `agent_app` DML grants required for feedback truth and prove `auth_app` and `commerce_app` remain denied.
- Keep retrieval/index implementation, sensitive PendingAction/ActionReceipt creation or execution, evaluation APIs/sandbox, memory summaries, handoff/tickets, and later behavior out of this slice.

## Acceptance criteria

1. `POST /api/chat/stream` enforces the frozen direct-user issuer, audience, type, principal permission/state, token-derived subject, persisted `X-Session-Id` ownership, and production evaluation-header denial before reading or creating turn truth.
2. Streaming and non-streaming routes share the same durable conversation/turn/correlation identity. Same-intent replay returns the existing outcome without another logical turn or tool execution; conflicting reuse and cross-session correlation reject.
3. The SSE egress filter emits only `token`, `action_receipt`, `done`, and `error`, with fixed public fields. Unknown internal event types and unknown/private fields fail closed and never pass through generically.
4. Public SSE never contains prompts, token claims, credentials, raw provider responses, routing/budget/circuit details, ToolSpecs, tool arguments or raw results, retrieval/evidence payloads, SQL details, stack traces, or internal exception text.
5. Ordinary text passes the configured bounded safety buffer before emission. Action-success language is withheld unless matching successful receipt truth exists; because CB-120/CB-121 are not yet implemented, CB-082 evidence cannot fabricate an `action_receipt` or an action-success claim.
6. Events preserve source order and exactly one terminal outcome. No event follows `done` or terminal `error`; internal retry/fallback remains server-side evidence and cannot create duplicate public answers or a false successful terminal event.
7. Disconnect, cancellation, timeout, and slow-consumer handling are bounded. They cannot roll back already committed durable truth, leave unbounded tasks, re-execute side effects, or disclose a private failure; a later authorized replay resolves from persisted state.
8. `POST /api/feedback` validates direct-user identity and persisted user/session/trace ownership, ignores no body identity authority, uses a server-generated stable feedback identity, and appends one authorized signal in `cs_db`. Same-intent retry is stable; conflicting key reuse, unknown trace/session, and cross-user access reject without another row.
9. Permission tests prove only required `agent_app` feedback-table DML, `auth_app`/`commerce_app` denial, runtime-DDL denial, and absence of migration/bootstrap or provider credentials in application configuration.
10. Deterministic end-to-end integration proves safe token streaming, filtered tool success/denial, provider failure, bounded retry/fallback visibility, budget exhaustion, replay, feedback, ordered durable evidence, and all identity/ownership rejection modes without a real provider.
11. No retrieval/index, sensitive PendingAction/ActionReceipt creation or execution, evaluation evidence/sandbox, memory summary, handoff/ticket, web, deployment, or later-slice behavior is introduced.

## Rejection paths

- Reject wrong issuer/audience/type, inactive or insufficient principal, missing/forged/cross-user support session, body identity substitution, production evaluation context, unknown trace, and cross-session correlation.
- Reject a second conversation or turn truth for the streaming route, replay that re-executes a tool, conflicting idempotency/correlation reuse, and public success before the durable terminal turn commits.
- Reject wildcard/generic forwarding of internal events or fields, raw provider/tool/retrieval/evidence output, prompts, tokens, credentials, database errors, stack traces, and internal exception text in SSE or feedback responses.
- Reject unknown public event names, events after a terminal event, duplicate terminals, reordered deltas, internal retries exposed as multiple answers, unbounded buffering, and unbounded work after disconnect.
- Reject an action-success token without matching successful receipt truth and any synthetic `action_receipt` created before its owning slices.
- Reject feedback with client-selected owner/trace truth, unknown or cross-user session/trace, conflicting key reuse, mutable overwrite, or feedback stored outside the authorized `cs_db` boundary.
- Reject capability claims based only on handler mocks without deterministic fake-provider, real JWT/session and `cs_db`, bounded-agent/tool, ordered SSE, disconnect/replay, feedback, permission-denial, redaction, and controlled-failure evidence.

## Out of scope

- retrieval, reranking, knowledge indexing, FAQ grounding, citation sufficiency, and Support Redis summary/memory behavior;
- PendingAction preparation/confirmation, ActionReceipt creation or commerce receipt truth, sensitive tool execution, and agent-side receipt projection/turn commit;
- evaluation identity/evidence APIs, sandbox lifecycle, human handoff/tickets, web UI, deployment, real provider integration, measured latency, and measured quality.
