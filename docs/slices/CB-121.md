## Goal

Complete the agent side of sensitive-action confirmation by binding one user confirmation to one
commerce-owned PendingAction, projecting only an authenticated immutable ActionReceipt into `cs_db`,
and preserving the successful commerce-owned ActionReceipt as the turn-level commit point and sole
public action-status carrier. Network or model retry after that commit may regenerate explanation
text but can never execute commerce again.

## In scope

- Persist a bounded agent-side pending-action reference bound to the server-owned support session,
  user, turn, trace, tool identity, canonical argument commitment, commerce PendingAction id, expiry,
  and evaluation sandbox where present. It is a reference, not action truth.
- Interpret confirmation only within the current owned support session and exact pending turn. Use a
  closed deterministic confirmation/decline grammar; ambiguous prose asks for clarification and
  never calls confirm. A model cannot assert confirmation, change arguments, choose an action id, or
  expand scope.
- Acquire a fresh exact-scope OBO only after the confirmation boundary succeeds, then call the fixed
  CB-120 confirm endpoint with the stored PendingAction identity. Revalidate direct-user ownership,
  session, turn, and sandbox context before the call.
- Validate the complete untrusted commerce receipt schema, identity, owner/session/turn correlation,
  action/argument commitment, authoritative result, and server signature/authenticated channel.
  Persist one immutable ActionReceipt projection plus turn evidence atomically in `cs_db`.
- Treat the successful commerce-owned ActionReceipt as the authoritative turn-level commit point.
  The atomically persisted local projection is only the publication/replay checkpoint that must
  precede public receipt emission; its failure cannot undo, delay, or redefine the commerce commit.
  Emit one public `action_receipt` event derived only from that projection, followed by optional
  bounded explanation `token` prose and one terminal event. Token text remains non-authoritative.
- On same-turn replay, disconnect, response loss, model failure, or process restart, read the stored
  projection first. If present and consistent, return/re-emit it without another commerce confirm.
  Regenerated explanation uses stored receipt truth and shares the existing attempt budget; it never
  re-enters the tool mutation boundary.
- Preserve CB-080 turn/event ordering, CB-081 budget/OBO/ToolSpec controls, CB-082 filtered SSE and
  receipt-only public status, and CB-120 commerce truth. `agent-service` never writes
  `commerce_db`, and its projection cannot override an ActionReceipt.
- Add only exact `agent_app` migration/runtime grants for the new `cs_db` reference/projection family
  and prove cross-database and unrelated-table denial.

## Acceptance criteria

1. Only the owner of the exact support session and pending turn can confirm or decline. Wrong
   issuer/audience/type, cross-user/session/turn, forged pending id, body owner, stale trace, sandbox
   mismatch, or non-ACTIVE/indeterminate evaluation context fails closed without a commerce call.
2. The deterministic confirmation boundary accepts only the documented positive forms, handles the
   documented declines without execution, and treats ambiguous or unrelated text as clarification.
   Model output, tool output, token prose, and a client-supplied boolean cannot establish consent.
3. The confirm request uses only the stored PendingAction identity and a freshly validated
   exact-scope OBO. The model/client cannot change action type, arguments, resource version, owner,
   session, scope, or sandbox after preparation.
4. Commerce responses are untrusted: malformed, duplicate-key, oversized, unknown-field,
   wrong-owner/session/turn/action/commitment, missing receipt, contradictory result, or unavailable
   responses do not create a projection or success event.
5. One accepted immutable ActionReceipt projection and its local publication/replay evidence are
   written atomically in `cs_db`. A controlled failure at every local write point leaves no partial
   receipt event or falsely published local checkpoint, while the successful commerce-owned
   ActionReceipt remains the already-committed authoritative turn result.
6. Same-intent replay, response loss after commerce commit, disconnect, and restart converge to one
   projection and one authoritative receipt identity. The agent resolves stored receipt truth before
   any retry and never invokes commerce twice for a committed action.
7. Conflicting receipt identity or content for the same action/turn fails closed. Missing local
   projection after an indeterminate commerce response is not evidence that the action did not
   execute; recovery re-queries the fixed CB-120 result boundary rather than re-preparing or changing
   intent.
8. Public streaming and non-streaming responses expose action status only through the fixed
   `action_receipt` schema. `token` prose cannot synthesize, replace, contradict, or be interpreted by
   clients as receipt truth. Receipt precedes any success explanation and the stream has one terminal
   outcome.
9. Post-commit model/network retry may regenerate bounded explanation from stored receipt data under
   the shared attempt budget. Exhaustion or provider failure cannot roll back the receipt, re-execute
   commerce, or change action status.
10. Direct declines, expiry, CB-120 rejection, OBO denial/unavailability, commerce timeout, malformed
    receipt, and persistence failure have distinct bounded outcomes and cannot be folded into
    success, absence, or a receipt event.
11. Exact grants allow only required `agent_app` access in `cs_db`; `agent_app` cannot access
    PendingAction, ActionReceipt, refund, order, or other `commerce_db` truth, and `commerce_app`/
    `auth_app` cannot mutate agent receipt projections.
12. No new commerce mutation, arbitrary action engine, memory/summary packing, handoff/ticket,
    failure-candidate export, web, deployment, real-provider credential, trace authority, or measured
    performance behavior is introduced.

## Rejection paths

- Reject confirmation from an unknown/cross-owner session, wrong turn/trace, expired or replaced
  pending reference, forged action id, changed argument commitment, wrong token mode/scope/actor, or
  sandbox mismatch/inactivity/unavailability.
- Reject model-decided consent, client booleans, broad natural-language inference, confirmation that
  is not bound to the exact current PendingAction, and any re-prepare/re-execute fallback after an
  indeterminate result.
- Reject permissive commerce JSON parsing, duplicate object keys, unbounded response collection,
  unknown fields, missing receipt identity, correlation/commitment mismatch, and local synthesis of
  a receipt.
- Reject treating local projection persistence as a second or delayed turn commit, public receipt
  emission before durable projection, partial event persistence, mutable/overwriteable projections,
  and same-key conflicting receipt content.
- Reject retry that calls commerce after a receipt projection exists, explanation failure that
  changes receipt state, token prose used as action status, multiple receipt events, or more than one
  terminal stream event.
- Reject direct cross-database access, broad grants, new asynchronous action carriers, generalized
  recovery scanners, and CB-130 or later work implemented early.

## Out of scope

- PendingAction validation/consume, business mutation, and authoritative ActionReceipt persistence
  owned by CB-120;
- new sensitive business actions, refund state changes, payment/provider behavior, settlement, and
  chargeback;
- summary watermarks, memory packing, prompt/PII and tiered output-safety changes owned by CB-130;
- authoritative handoff tickets and `HUMAN_PENDING` behavior owned by CB-131;
- failure-candidate capture/export, web UI, observability expansion, deployment, real-provider access,
  and measured load/latency/quality.

## Required evidence

- executable confirmation/decline/ambiguity grammar tests and real owner/session/turn/sandbox
  authorization evidence proving no model/client consent shortcut;
- real CB-120 integration for fresh exact-scope OBO, same-intent confirmation, conflict/rejection,
  response loss after commit, dependency unavailability, restart, and no second commerce execution;
- strict bounded receipt-decoder evidence for duplicate keys, unknown/missing/wrong-typed fields,
  oversized payloads, identity/commitment mismatch, and authenticated source;
- real MySQL `cs_db` evidence for atomic projection/publication-checkpoint evidence, rollback at each
  local write boundary without changing the authoritative commerce commit, immutable uniqueness,
  concurrent replay, and recovery after partial local failure;
- streaming/non-streaming evidence proving one receipt event from stored truth, receipt-before-success
  prose, token non-authority, one terminal event, disconnect/replay safety, and explanation-only retry
  under the shared attempt budget;
- exact runtime grant and cross-database denial evidence for `agent_app`, `commerce_app`, and
  `auth_app`;
- regressions for CB-080/081/082 turn ordering, budget, OBO, SSE, feedback, and public
  action-status contracts;
- evidence that no commerce truth mutation, memory, handoff, export, web, deployment, provider,
  observability authority, performance, or later behavior was implemented early;
- passing real `make ci` and exact additional integration commands recorded in the pull request;
- real Branch, PR, Commits, Tests, and Notes only at completion.
